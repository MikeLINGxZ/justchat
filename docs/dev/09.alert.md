# 信息提示（Alert）需求文档

## 目的

统一应用内的提示展示规范（错误、信息、可操作提示等），明确交互与视觉预期、聚合/去重规则、时长策略、可访问性和国际化要求，以便前端与产品在实现前对行为达成一致。

## 适用范围

- 应用内所有短时提示场景（非聊天流），包括但不限于：设置变更、导入导出结果、同步状态、网络/权限错误、第三方回调提示等。

## 用户场景

- 用户执行操作出现错误，需要展示可理解的错误标题与可展开的详细信息，以便用户知晓问题并执行后续操作（如重试、查看日志、联系支持）。
- 系统/后台产生信息性提示（例如：已保存、同步完成、导入成功），需短时提示并在内容重复时显示计数。
- 提示可包含直接操作（确认、取消、跳转到相关页面），并在必要时要求二次确认以防误操作。

## 功能性需求（以用户为中心）

1) 错误提示（Error）

- 展示要点：明显的错误视觉（警示色）、简短明确的错误标题、可查看/展开的详细信息（可包含错误码与建议采取的下一步）。
- 行为预期：错误提示默认不会短时间内自动消失（需用户操作或明确的超时策略），并提供常用辅助操作（例如：复制详情、重试、打开帮助链接）。

2) 普通信息提示（Info / Success / Warning）

- 展示要点：中性或成功色系，显示可选标题与主要内容；信息应简短、可理解。
- 时长与计数：默认短时可见（例如 5s），允许配置最长显示时长（不得超过 15s）；若收到完全相同的提示，多次出现应以计数形式呈现（显示为 "x{n}"），并在重复时重置该提示的显示时长。

3) 可操作提示（Actionable）

- 能在提示中提供明确的后续操作（例如确认/取消、查看详情、跳转），最多提供一到两个主要操作按钮以保持界面简洁。
- 对于具有破坏性或不可逆的操作，提示仅作为引导，实际执行需二次确认（模态或显式确认步骤）。

4) 聚合与并列显示策略

- 对于内容完全相同的提示，采用聚合策略显示计数，避免重复堆叠干扰用户体验。
- 同一时刻并列展示的提示数量应受限（建议不超过 3 个），并在数量超限时采用明确的优先级或时间策略处理旧提示。

5) 关闭与生命周期

- 用户可手动关闭任何可关闭提示；提示自动关闭需明确显示原因（超时或程序触发）。
- 关闭行为应触发可追踪的事件（例如：用户关闭、超时关闭、程序移除），以便统计与埋点。

6) 展现位置与适配

- 支持常见位置（例如右下角为默认），也允许针对重要信息采用页面顶部横幅样式。
- 移动端与桌面端应有一致的交互语义（移动端支持滑动关闭，桌面支持点击/键盘操作）。

## 非功能性需求

- 无障碍：提示需要对屏幕阅读器友好（根据提示重要性使用适当的播报优先级），可通过键盘访问与操作。
- 国际化：所有提示文本需走 i18n 管线，按钮与时间格式需支持本地化。
- 可配置性：产品可配置默认显示时长、并列上限与重复聚合策略（但默认值必须合理以保证良好体验）。
- 可追踪性：提示的创建、显示、点击操作与关闭事件应可埋点用于统计与分析。
- 高复用性：封装为组件，可以在前端代码任意地方调用；后端可以通事件调用。

## 验收标准（Acceptance Criteria）

1. 错误提示以警示色明显呈现，包含易懂的错误标题与可查看的详细信息；用户能复制或查看详情用于问题排查。
2. 普通提示在约 5s 后自动消失（除非带操作），相同内容的重复提示以计数方式合并且每次重复会重置显示时长；最长显示时间不超过 15s。
3. 可操作提示的动作清晰（至多两个主要动作），操作触发需能区分是否自动关闭提示；危险操作需有二次确认流程。
4. 同时显示的提示不超过预定上限，超出时按优先级或时间顺序处理旧提示，界面无重叠或遮挡问题。
5. 屏幕阅读器能正确播报新提示（高优先级错误应更高优先级播报）；所有交互可通过键盘完成。
