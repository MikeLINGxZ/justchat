## 二、各库详细分析

### 2.1 eino (cloudwego/eino)

**维护方**：字节跳动 CloudWeGo 团队

**定位**：Go 语言 LLM/AI 应用开发框架，参考了 LangChain、Google ADK 等设计。

**核心架构**：
- **Components（组件抽象）**：ChatModel、Tool、Retriever、Embedding、ChatTemplate 等，官方通过 [eino-ext](https://github.com/cloudwego/eino-ext) 提供 OpenAI、Ollama、Claude、Gemini 等实现
- **ADK（Agent Development Kit）**：Agent 构建能力，包括工具使用、多 Agent 协调、上下文管理、中断/恢复
- **Compose（编排）**：基于 Graph 的工作流编排，可将 Graph 暴露为 Tool 供 Agent 调用

**核心功能**：

| 功能领域 | 能力 |
|---|---|
| Agent | ChatModelAgent（ReAct 循环内置）、DeepAgent（任务分解+子 Agent 委派+进度跟踪） |
| 多 Agent | DeepAgent 支持配置多个子 Agent 协作 |
| 工具调用 | 完整支持，内置 shell、python、web search 等工具 |
| 工作流/Graph | compose.NewGraph() 构建有向图，支持条件分支、Lambda 节点、ChatModel 节点 |
| 流式处理 | 自动处理流式数据在编排中的传递、合并、复制 |
| 回调/切面 | OnStart/OnEnd/OnError/OnStartWithStreamInput/OnEndWithStreamOutput |
| 中断/恢复 | **原生支持**，Agent/Tool 可暂停等待人工输入，从检查点恢复（Human-in-the-Loop） |
| 记忆 | 通过上下文管理实现，未提供独立的 Memory 服务抽象 |
| 评估 | [eino-ext](https://github.com/cloudwego/eino-ext) 包含评估器和 Prompt 优化器 |
| 可视化调试 | [eino-devops](https://github.com/cloudwego/eino-ext/tree/main/devops) 提供可视化开发调试 |

**生态**：
- 官方仓库：eino（核心）、eino-ext（组件实现）、eino-examples（示例）、eino-devops（可视化）
- 中文文档完善：[www.cloudwego.io/docs/eino/](https://www.cloudwego.io/docs/eino/)
- 社区：飞书用户群，Issues 活跃
- 组件实现丰富：OpenAI、Claude、Gemini、Ark、Ollama、Elasticsearch 等

### 2.2 langchaingo (tmc/langchaingo)

**维护方**：社区驱动（原 tmc 维护，现逐步转向社区维护）

**定位**：LangChain 的 Go 语言实现，以"组合性"构建 LLM 应用。

**核心模块**：

| 模块 | 功能 |
|---|---|
| `llms` | LLM 抽象层（OpenAI、Gemini、Ollama 等） |
| `agents` | Agent 实现（ReAct 循环等） |
| `chains` | 链式调用（ConversationChain、LLMChain 等） |
| `memory` | 会话记忆（Buffer、Summary 等） |
| `tools` | 工具定义与调用 |
| `vectorstores` | 向量数据库集成（Milvus、Weaviate、Pinecone 等） |
| `embeddings` | 嵌入模型抽象 |
| `documentloaders` | 文档加载器（PDF、HTML 等） |
| `prompts` | Prompt 模板 |
| `outputparser` | 输出解析器 |

**核心功能**：

| 功能领域 | 能力 |
|---|---|
| Agent | 基于 ReAct 循环的传统 Agent，需自行编写循环逻辑 |
| 多 Agent | 无原生支持 |
| 工具调用 | 完整支持，通过 `tools` 包定义 |
| 工作流/Graph | 无原生 Graph 编排，主要用 Chains 链式调用 |
| 流式处理 | 支持 |
| 回调 | 支持 callback 机制 |
| 中断/恢复 | **无原生支持** |
| 记忆 | `memory` 包提供 Buffer、Summary 等记忆类型，但无自主判断机制 |
| RAG | 完整链路：DocumentLoader → TextSplitter → Embedding → VectorStore → Retriever |
| 评估 | 无独立评估框架 |

**生态**：
- 作为 LangChain 生态的 Go 移植，继承了 LangChain 的设计理念和组件抽象
- 组件集成最广：OpenAI、Google Gemini、Ollama、HuggingFace 等 LLM；多种 VectorStore
- 文档：[tmc.github.io/langchaingo/docs/](https://tmc.github.io/langchaingo/docs/)
- Discord 社区
- 版本号仍停留在 v0.1.x，更新节奏较慢

### 2.3 adk-go (google/adk-go)

**维护方**：Google

**定位**：代码优先的 Go Agent 开发工具包，用于构建、评估和部署 AI Agent。

**核心模块**：

| 模块 | 功能 |
|---|---|
| `agent` | Agent 核心抽象与实现 |
| `model` | LLM 模型接口 |
| `tool` | 工具系统（Function Tool、MCP 等） |
| `memory` | 记忆服务 |
| `runner` | Agent 运行器，管理执行流程 |
| `session` | 会话状态管理 |
| `server` | HTTP 服务（A2A 互操作等） |
| `telemetry` | 遥测/追踪 |
| `plugin` | 插件机制 |

**核心功能**：

| 功能领域 | 能力 |
|---|---|
| Agent | code-first 定义，支持工具使用、子 Agent 委派 |
| 多 Agent | 模块化多 Agent 组合 |
| 工具调用 | 支持自定义 Function Tool、MCP 集成 |
| 工作流/Graph | 通过 Agent 组合实现，无独立 Graph/Workflow 引擎 |
| 流式处理 | 支持 |
| 回调/切面 | 通过 plugin 机制实现 |
| 中断/恢复 | 无显式的 Human-in-the-Loop 机制 |
| 记忆 | `memory` 模块，提供记忆服务 |
| 评估 | 未在 README 中强调 |
| 部署 | 容器化友好，Cloud Run 支持 |

**生态**：
- Google 官方维护，Apache 2.0 许可
- 多语言家族：Python ADK、Java ADK、Go ADK + [ADK Web](https://github.com/google/adk-web) 可视化
- 优化 Gemini 但模型无关
- 文档：[google.github.io/adk-docs/](https://google.github.io/adk-docs/)
- 示例丰富：`examples/` 目录涵盖多 Agent、工具、MCP 等

### 2.4 trpc-agent-go (trpc-group/trpc-agent-go)

**维护方**：腾讯 tRPC 团队

**定位**：面向 tRPC 生态的全能型 Go Agent 框架，功能覆盖面最广。

**核心模块**（最丰富）：

| 模块 | 功能 |
|---|---|
| `agent` | LLMAgent、ChainAgent、ParallelAgent、CycleAgent、GraphAgent |
| `model` | 多 LLM 支持（OpenAI、DeepSeek 等） |
| `tool` | Function Tool、MCP Tool、DuckDuckGo Tool 等 |
| `memory` | 内存/Redis 记忆服务，含 CRUD、搜索 和 Tool 集成 |
| `knowledge` | RAG 知识检索 |
| `session` | 会话状态与事件管理 |
| `runner` | Agent 执行器，管理 Session/Memory |
| `planner` | 分层规划器 |
| `graph` | **GraphAgent**：类型安全的图工作流，支持多条件分支路由，功能等同于 Go 版 LangGraph |
| `skill` | Agent Skills：基于 `SKILL.md` 的可复用工作流，安全执行 |
| `artifact` | 版本化文件存储（内存、S3、COS） |
| `evaluation` | 评估框架：eval sets + 可插拔指标 |
| `server` | Gateway、AG-UI、A2A 协议支持 |
| `telemetry` | OpenTelemetry 追踪与指标 |
| `codeexecutor` | 安全代码执行 |
| `plugin` | 插件机制 |

**核心功能**：

| 功能领域 | 能力 |
|---|---|
| Agent | 内置多种 Agent：LLMAgent、ChainAgent、ParallelAgent、CycleAgent、GraphAgent |
| 多 Agent | **最完善**：链式(Chain)、并行(Parallel)、循环(Cycle)、图(Graph)多 Agent 编排 |
| 工具调用 | 完整支持：Function Tool、MCP、DuckDuckGo、文件工具、Agent 作为 Tool |
| 工作流/Graph | **GraphAgent** 类型安全图工作流+多条件路由，功能相当于 LangGraph Go 版 |
| 流式处理 | 完整支持 |
| 回调/切面 | 全链路 OpenTelemetry 追踪 |
| 中断/恢复 | **支持**：提供 Human-in-the-Loop 示例，支持 context 取消、按 requestID 取消 |
| 记忆 | **完整记忆系统**：内存/Redis 后端，自主 CRUD+搜索，记忆工具集成到 Agent |
| RAG | 完整 knowledge 模块：加载→嵌入→检索 |
| 评估 | 独立 evaluation 框架 |
| 技能系统 | Agent Skills（SKILL.md），支持安全隔离执行 |
| UI 集成 | AG-UI 协议（SSE）+ CopilotKit、TDesign Chat 客户端 |
| A2A | Agent-to-Agent 互操作协议 |

**生态**：
- 腾讯官方维护，已在腾讯元宝、腾讯视频、腾讯新闻、IMA、QQ音乐等业务落地验证
- 版本迭代极快：v1.9.1（114 个 release）
- 文档完善：[trpc-group.github.io/trpc-agent-go/](https://trpc-group.github.io/trpc-agent-go/)
- 示例最丰富：覆盖 Tool、Memory、Knowledge、Graph、MCP、AG-UI、Evaluation、Skill、Artifact、A2A 等

## 三、核心需求匹配分析

### 需求 1：具备对话能力，模型自主决定直接回答、交给专业 Agent 回答或使用工作流

| 库 | 直接回答 | 委派专业 Agent | 工作流执行 | 评估 |
|---|---|---|---|---|
| **eino** | ChatModelAgent 自带 ReAct | DeepAgent 支持子 Agent 委派 | compose.Graph 编排，且 Graph 可暴露为 Tool | **优秀** - 三层决策模型完整 |
| **langchaingo** | 通过 chains 实现 | 无原生支持 | 仅 Chains 链式，无 Graph | **不足** - 缺少 Agent 委派和工作流 |
| **adk-go** | code-first Agent | 多 Agent 组合支持 | 通过 Agent 组合实现，无独立工作流引擎 | **良好** - 缺少独立工作流引擎 |
| **trpc-agent-go** | LLMAgent 完整实现 | Chain/Parallel/Cycle/Graph Agent | **GraphAgent** 功能最完善 | **优秀** - 多 Agent + Graph 双引擎 |

### 需求 2：具备记忆潜力，自主判断是否记住，按需回忆

| 库 | 记忆存储 | 自主判断 | 回忆唤醒 | 评估 |
|---|---|---|---|---|
| **eino** | 上下文管理，无独立 Memory 服务 | 无 | 无 | **不足** - 需自行实现 |
| **langchaingo** | memory 包：Buffer/Summary | 无自主判断 | 窗口/摘要方式 | **基础** - 有记忆模块但无智能判断 |
| **adk-go** | memory 模块 | 未明确 | 未明确 | **基础** - 记忆模块存在但功能有限 |
| **trpc-agent-go** | memory 服务（内存/Redis）+ Tool 集成 | **记忆工具集成到 Agent，Agent 自主决策** | 搜索+CRUD，Agent 按需调用记忆 Tool | **优秀** - 唯一提供 Agent 自主记忆决策的框架 |

### 需求 3：工具调用，敏感工具可拦截用户确认

| 库 | 工具调用 | 敏感工具拦截 | 用户确认流程 | 评估 |
|---|---|---|---|---|
| **eino** | 完整支持 | **原生 Interrupt/Resume 机制** | Human-in-the-Loop 支持暂停→人工确认→恢复 | **优秀** - 唯一原生实现中断/恢复 |
| **langchaingo** | 完整支持 | 无 | 无 | **不足** - 需自行实现 |
| **adk-go** | 完整支持 | 无显式机制 | 无 | **不足** - 需自行实现 |
| **trpc-agent-go** | 完整支持 | 支持 Human-in-the-Loop | 提供示例但非框架级原生机制 | **良好** - 有示例和 Context 取消，但不如 eino 优雅 |

## 五、推荐方案

### 首选推荐：**trpc-agent-go**

**核心理由**：
1. **功能最全面**：是唯一同时满足三个核心需求的框架
   - 多 Agent 委派 + GraphAgent 工作流 → 满足需求 1
   - 完整记忆服务 + Tool 集成让 Agent 自主决策记忆 → 满足需求 2
   - 有 Human-in-the-Loop 基础和示例 → 可满足需求 3
2. **生产验证**：腾讯多个亿级业务线已落地使用
3. **生态最完整**：MCP、AG-UI、A2A、Skills、Evaluation 一应俱全
4. **中文文档+社区**：降低接入成本
5. **版本迭代快**：活跃开发中

**需要自行补充的部分**：
- 敏感工具拦截确认流程需要基于 Human-in-the-Loop 示例进行深度定制（eino 在此方面更原生）

### 次选：**eino**

**适用场景**：如果 Human-in-the-Loop（敏感工具拦截）是你的**最高优先级**需求，eino 的原生 Interrupt/Resume 机制是最优雅的解决方案。

**权衡**：
- eino 的记忆系统较薄弱，需要自行实现"自主判断是否记忆"的机制
- eino 的工作流引擎（compose.Graph）比 trpc-agent-go 的 GraphAgent 更轻量化，但功能略少

### 补充参考：**langchaingo**

适合快速原型和简单对话场景，但不适合需要 Agent 协作、记忆智能、工作流编排的复杂 AI Chat Box。

### 补充参考：**adk-go**

Google 官方出品，适合偏好 Google 生态（Gemini、Cloud Run）的团队。功能介于 eino 和 trpc-agent-go 之间，但作为独立的 Agent 框架，缺少工作流引擎和 Human-in-the-Loop 原生支持。

---

## 六、混合使用可能性

eino 和 trpc-agent-go 可以**互补使用**：
- **trpc-agent-go** 作为主框架，提供 Agent 生命周期管理、记忆、多 Agent 编排、Graph 工作流
- **eino** 的 Interrupt/Resume 机制可作为参考设计，在 trpc-agent-go 上实现更精细化的敏感工具拦截

或者采用分层架构：
- 上层 Agent 逻辑：使用 trpc-agent-go 的 LLMAgent/GraphAgent
- 工具执行层：对于需要用户确认的敏感 Tool，自行实现拦截中间件（参考 eino 的 Interrupt 设计模式）
