# trpc-agent-go 记忆能力调研报告

## 调研目的

对比 trpc-agent-go 框架的记忆模块与 `11.memory_core.md` 设计文档，评估框架能力覆盖度，明确哪些能力可直接复用、哪些需自建、哪些需改造。

---

## 1. trpc-agent-go 记忆模块概览

### 1.1 存储后端

| 后端 | 搜索模式 | 适用场景 |
|------|----------|----------|
| InMemory | 关键词 (BM25) | 开发测试 |
| Redis | 关键词 | 生产级扩展 |
| SQLite | 关键词 | 嵌入式/单节点 |
| SQLiteVec | 向量 + 混合搜索 | 嵌入式 + 语义搜索 |
| MySQL | 关键词 + JSON | 关系型生产环境 |
| MySQLVec | 向量 + 混合搜索 | MySQL + 语义搜索 |
| PostgreSQL | 关键词 + JSONB | 关系型生产环境 |
| pgvector | 向量 + 混合搜索 | PostgreSQL + 语义搜索 |

### 1.2 核心接口

```go
type Service interface {
    AddMemory(ctx, userKey, memory string, topics []string, opts ...) error
    UpdateMemory(ctx, memoryKey, memory string, topics []string, opts ...) error
    DeleteMemory(ctx, memoryKey) error
    ClearMemories(ctx, userKey) error
    ReadMemories(ctx, userKey, limit int) ([]*Entry, error)
    SearchMemories(ctx, userKey, query string, opts ...) ([]*Entry, error)
    Tools() []tool.Tool
    EnqueueAutoMemoryJob(ctx, sess *session.Session) error
    Close() error
}
```

### 1.3 数据模型

```go
type Memory struct {
    Memory      string     `json:"memory"`       // 记忆文本内容
    Topics      []string   `json:"topics"`        // 主题标签
    LastUpdated *time.Time `json:"last_updated"`
    Kind        Kind       `json:"kind"`          // "fact" 或 "episode"
    EventTime   *time.Time `json:"event_time"`    // 事件时间
    Participants []string  `json:"participants"`  // 参与者
    Location    string     `json:"location"`      // 地点
}

type Entry struct {
    ID        string    `json:"id"`
    AppName   string    `json:"app_name"`
    Memory    *Memory   `json:"memory"`
    UserID    string    `json:"user_id"`
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`
    Score     float64   `json:"score,omitempty"`
}
```

记忆 ID 通过 `SHA256(内容 + app + user + kind + event_time + participants + location)` 确定性生成，相同内容幂等覆盖。topics 不参与 ID 计算，修改标签不产生新记忆。

### 1.4 两套工具模式

| 模式 | 工具暴露 | 写入策略 |
|------|----------|----------|
| **Agentic 模式** | 6 个工具全部可用：`memory_add`、`memory_update`、`memory_search`、`memory_load`、`memory_delete`、`memory_clear` | LLM 自主决定读写，适合简单场景 |
| **Auto 模式** | 仅暴露 `memory_search` 给 Agent；写入工具隐藏，由后台 Extractor 自动完成 | 异步自动提取，适合生产场景 |

Agentic 模式默认启用 `add/update/search/load`，`delete/clear` 默认禁用。Auto 模式默认启用 `add/update/delete` 给 Extractor，`search` 暴露给 Agent。

### 1.5 Auto 自动记忆提取

```go
type MemoryExtractor interface {
    Extract(ctx context.Context, messages []model.Message,
        existing []*memory.Entry) ([]*Operation, error)
    ShouldExtract(ctx *ExtractionContext) bool
}
```

- 通过独立 LLM 模型分析用户消息和已有记忆，决定新增/更新/删除
- 支持消息数量阈值触发 (`CheckMessageThreshold`)、时间间隔触发 (`CheckTimeInterval`)
- 支持 AND/OR 组合触发条件
- 异步 Worker 执行：可配置并发数、队列大小、超时

### 1.6 搜索策略

- **关键词搜索**（非向量后端）：BM25 加权 + CJK 分词（gse）+ 英文分词 + 停用词过滤
- **向量搜索**（向量后端）：通过 embedding 模型做语义搜索
- **混合搜索**：RRF (Reciprocal Rank Fusion) 融合向量 + 关键词结果
- **去重**：Jaccard 相似度 >80% 的近重复结果只保留最高分项
- **Kind fallback**：当 kind 过滤结果 <3 条时自动回退到全量搜索
- **时间范围过滤**：支持 `TimeAfter`/`TimeBefore` 限定搜索的时间区间
- **单个用户记忆上限**：默认 1000 条，可配置

### 1.7 Runner 集成

```
Runner.Run() → 传递 MemoryService 到 Agent 上下文
            → LLM Agent 通过 Tools() 使用记忆工具
            → 响应完成后，Auto 模式触发 EnqueueAutoMemoryJob()
            → 异步 goroutine 执行 Extractor.Extract()
            → 执行 add/update/delete 操作
```

## 2. 与 11.memory_core.md 设计的对比

### 2.1 架构对比

| 维度 | trpc-agent-go | 11.memory_core.md |
|------|---------------|-------------------|
| **分层设计** | Service 接口 + Extractor + Backend | 存储层 / 检索层 / 记忆代理层 / 主聊天集成层 / 管理生命周期层（五层） |
| **两阶段读写** | 支持（Search 在前，Extract 在后） | 支持（检索在前，编码在后） |
| **core memory + retrieval memory** | 无明确区分，memory 统一为检索型 | 明确区分 core memory（注入 system prompt）和 retrieval memory（按需召回） |
| **Memory Agent** | 无独立 Agent，通过 Extractor 接口实现 | 有独立 Memory Agent 负责读写语义判断 |
| **生命周期管理** | 无内置生命周期管理 | 有巩固、矛盾检测、遗忘、可恢复性 |
| **Session 关联** | Session 独立模块，与 memory 解耦 | 无 session 层，直接管理记忆 |

### 2.2 数据模型对比

| 字段 | trpc-agent-go | 11.memory_core.md | 说明 |
|------|---------------|-------------------|------|
| id | 确定性 SHA256 生成 | 自增主键 | 框架的幂等设计 vs 自有设计 |
| content | `Memory.Memory` | `content` | 对应 |
| summary | 无独立摘要字段 | `summary`（标题） | **框架缺失** |
| type / kind | `Kind`: fact / episode | `type`: fact / information / event | 框架少了 `information`，`episode` ≈ `event` |
| target | 无（统一 user 维度） | `target`: user / memory | **框架缺失**，无法区分用户记忆和助手记忆 |
| topics | 有，字符串数组 | 无 | 自有设计无此字段 |
| embedding_id | 后端内置，不暴露 | 独立关联字段 | 自有设计可独立管理 |
| is_forgotten / recall_count / last_recalled_at / last_used_at | 无 | 有 | **框架缺失**，无法做生命周期管理 |
| participants / location / event_time | 有（episode 元数据） | 不独立，写入 content | 框架更结构化 |
| char_count | 无 | 有 | 框架未做容量控制 |

### 2.3 记忆类型覆盖

| 自有设计 | trpc-agent-go | 覆盖？ |
|----------|---------------|--------|
| 用户记忆 (target=user) | `UserKey.UserID` 维度 | 部分覆盖，但无 target 字段区分 |
| 助手记忆 (target=memory) | 不支持 | **不支持** |
| fact 类型 | 支持 (`KindFact`) | 覆盖 |
| information 类型 | 不支持 | **不支持**（需用 fact 模拟） |
| event 类型 | 支持 (`KindEpisode`) | 覆盖 |
| 手动记忆 (source=manual) | 不支持 source 字段 | **不支持** |
| 软删除 / 遗忘恢复 | PostgreSQL/MySQL 支持软删除 | 部分覆盖 |

### 2.4 工作流程对比

| 流程步骤 | trpc-agent-go | 11.memory_core.md | 匹配度 |
|----------|---------------|-------------------|--------|
| 启动初始化 | NewService + 配置 | 初始化 DB + 迁移 + 检索器 + 生命周期 | 自有设计更完善 |
| 聊天前注入 core memory | **不支持** | 核心记忆注入 system prompt | **gap** |
| 聊天前检索记忆 | SearchMemories | Hybrid Searcher 检索 | 覆盖 |
| 聊天后异步编码 | EnqueueAutoMemoryJob + Extractor | Memory Agent 异步编码 | 覆盖 |
| 关键词提取 | 内置于搜索中 | 独立关键词提取步骤 | 框架内聚更好 |
| 多模态保护 | **不支持** | 有明确禁止规则 | **gap** |
| embedding 回填 | 需手动管理 | 设计为后台补齐 | 部分覆盖 |
| 生命周期维护 | **不支持** | 巩固/矛盾/遗忘 | **gap** |

### 2.5 搜索能力对比

| 能力 | trpc-agent-go | 11.memory_core.md |
|------|---------------|-------------------|
| FTS5 / LIKE | 无（仅向量后端的 SQL 查询，非 FTS） | 核心依赖 |
| 向量搜索 | 有（sqlitevec/mysqlvec/pgvector） | 渐进增强能力 |
| 混合搜索 (RRF) | 有 | 设计中有 |
| CJK 分词 | 有（gse 分词器） | 未指定，仅标点/空格切分 |
| 时间范围过滤 | 有（TimeAfter/TimeBefore） | 无 |
| 去重 | 有（Jaccard 80%） | 无（依赖 Memory Agent） |
| Kind 回退 | 有（<3 条自动回退） | 无 |
| 兜底策略 | 无 | 返回少量最重要记忆 |
| embedding provider 配置 | 内置支持 | 设计为设置项 |

### 2.6 缺失能力清单

以下为 11.memory_core.md 设计中有但 trpc-agent-go 不直接支持的能力：

| 能力 | 重要程度 | 说明 |
|------|----------|------|
| **Core Memory 注入 system prompt** | 高 | 框架只管检索召回，不做 core memory 注入 |
| **target=memory（助手记忆）** | 高 | 框架仅支持 user 维度，无法区分用户画像和系统约定 |
| **memory type=information** | 中 | 框架仅 fact/episode，缺少可变化的偏好/状态类型 |
| **source 字段追踪** | 中 | 无法区分 agent 写入、手动写入、legacy 迁移 |
| **字符数容量控制** | 中 | 无 char_count、容量上限检查 |
| **Summary 标题字段** | 中 | 框架缺少 summary 独立字段 |
| **生命周期管理** | 中 | 巩固、遗忘、矛盾检测需自建 |
| **is_forgotten / recall_count / last_recalled_at / last_used_at** | 中 | 软删除部分支持，其余无 |
| **Memory Agent 独立代理** | 中 | 框架用 Extractor 替代，无专用 Agent |
| **session_search 工具** | 低 | 需自行对接 session 模块 |
| **多模态输入保护** | 低 | 需在 Extractor 外层拦截 |
| **embedding 配置页 / 补齐策略** | 低 | 框架有配置能力但无管理 UI |

## 3. trpc-agent-go 独有的能力（自有设计未覆盖）

| 能力 | 说明 |
|------|------|
| **确定性 ID（幂等写入）** | SHA256 生成 ID，相同内容自动覆盖，不会重复 |
| **Topics 标签系统** | 每条记忆可打多标签，修改标签不产生新记忆 |
| **Auto 自动提取模式** | 通过独立 LLM 异步分析会话、自动生成操作，无需模型主动调用工具 |
| **触发策略（Checker）** | 消息数量阈值 / 时间间隔 / AND/OR 组合，灵活控制提取频率 |
| **Kind 自动回退** | Kind 过滤结果不足时自动回退到全量搜索 |
| **Episodic 时序过滤** | 按时间范围搜索事件记忆 |
| **结构化 Episodic 元数据** | participants、location、event_time 独立字段 |
| **RRF 混合搜索** | 自带向量+关键词融合排序 |
| **多存储后端** | 8 种后端即插即用，无需自己实现存储层 |
| **Extractor 独立模型** | 记忆提取模型可与聊天模型不同 |

## 4. 集成建议

### 4.1 可直接复用的能力

1. **存储层**：直接使用 `sqlite` 或 `mysql` 后端，节省大量 CRUD 开发
2. **搜索能力**：关键词搜索 + CJK 分词 + RRF 混合搜索 + 去重
3. **Tools**：6 个标准工具可直接注册给 Agent
4. **Auto 提取**：Extractor 接口 + Worker 异步处理框架
5. **Runner 集成**：`WithMemoryService` 标准挂载

### 4.2 需要自建的能力

1. **Core Memory 注入**：在聊天前自行读取记忆中 `target` 区分后的条目，渲染为 system prompt 文本注入
2. **数据模型扩展**：通过自定义字段扩展 `Memory` 结构，增加：
   - `summary` 标题字段
   - `target`（user / memory）
   - `type` 扩展（fact / information / event）
   - `source`、`char_count`、`is_forgotten`、`recall_count`、`last_recalled_at`、`last_used_at`
   - 实现方式：在 Topics 数组或自定义 JSON 字段中携带，或 fork 后端添加字段
3. **容量控制**：在 `AddMemory` 前检查 `target` 维度的总字符数，超限时拒绝或触发替换逻辑
4. **Memory Agent**：在 Extractor 外层包装为 Memory Agent，传入 core memory 快照 + 用户消息，执行 replace/no-op 判断
5. **生命周期管理**：自建后台定时任务，实现巩固、矛盾检测、遗忘
6. **多模态保护**：在 `EnqueueAutoMemoryJob` 前检查消息是否包含附件/外部素材引用
7. **session_search 工具**：对接 session 模块实现历史会话搜索
8. **管理页面 API**：基于 Service 接口包装管理端所需的 CRUD、统计、筛选接口

### 4.4 推荐实现路径

**第一期（MVP）可基于 trpc-agent-go 直接完成：**

| 模块 | 实现方式 |
|------|----------|
| 存储 CRUD | 使用 `sqlite` 后端 |
| 记忆检索 | 使用 `SearchMemories` + 关键词搜索 |
| 自动编码 | 使用 Extractor + Auto 模式 |
| Agent 工具 | 使用 `memory_search`、`memory_add`、`memory_update` |
| Core memory 注入 | 自建 Renderer，读取记忆后注入 system prompt |

**第二期增加：**

| 模块 | 实现方式 |
|------|----------|
| target 区分 (user/memory) | 扩展数据模型，在 Topics 或自定义字段中标记 |
| 容量控制 | 自建检查逻辑 |
| 生命周期管理 | 自建后台定时任务 |
| 向量搜索 | 切换为 `sqlitevec` 后端 |

## 5. 结论

trpc-agent-go 的记忆模块覆盖了 `11.memory_core.md` 设计中约 **60%** 的核心能力，包括：存储 CRUD、关键词和向量搜索、异步自动提取、Agent 工具集成。这些能力可直接复用，减少大量基础开发工作。

**需要通过自建扩展层补齐的关键能力（按优先级排序）：**

1. Core Memory 注入 system prompt（无此能力则记忆系统不完整）
2. target 字段区分用户记忆 / 助手记忆
3. type=information 类型支持
4. 字符数容量控制
5. Summary 标题字段
6. 生命周期管理（巩固、遗忘、矛盾检测）
7. 多模态输入保护

建议采用"框架复用 + 自建扩展层"的混合架构，在 trpc-agent-go 的 Service 接口之上构建扩展层，以最小开发成本实现完整设计。
