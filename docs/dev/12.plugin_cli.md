# CLI 插件

继 MCP 工具接入完成后，本期为「插件与工具」体系新增第三种扩展形态：**CLI 插件**。它把系统已有的命令行工具（如飞书 CLI、`gh`、`aws` 等）包装成 AI 可调用的工具，让用户以最低成本接入 CLI 生态。

> 实施时同时遵循 `docs/dev/00.rules.md`。

---

## 1. 范围与定位

* **目标**：让用户能把 npm 上的任意 CLI 工具（或本地一个 `package.json` 目录）作为插件接入 Lemontea，由 AI 通过统一的工具协议调用。
* **不在本期范围**：
  * 不要求 CLI 作者做任何适配，零侵入接入。
  * 不实现 CLI 作为「用户直接使用的命令面板/嵌入式终端」，仅作为 AI 工具暴露。
  * 不引入新的运行时；复用 onboarding 已下载的 Node.js（即「插件运行时」）执行 `npm` 命令。

* **与现有扩展的关系**：
  * 复用 `data_models.ExtensionItem` 数据模型，新增 `kind: "cli"`，与 `mcp`、`plugin` 并列。
  * 复用 `service/plugin` 的列表 / 启用 / 禁用 / 删除流程。
  * 复用现有「插件与工具」设置页（左列表 + 右详情）。

## 2. 数据目录布局

本期同时整理整个扩展系统的数据目录布局：所有扩展统一收纳到 `{data_dir}/plugins/` 下。

```
{data_dir}/plugins/
├── cli/{name}/                  # CLI 插件包内容（覆盖式安装，单版本）
│   └── node_modules/...
├── cli_data/{name}/             # CLI 持久化数据，跨升级/重装保留
│   ├── home/                    → HOME / USERPROFILE
│   ├── config/                  → XDG_CONFIG_HOME
│   ├── data/                    → XDG_DATA_HOME
│   ├── cache/                   → XDG_CACHE_HOME
│   └── manifest.json            # AI 生成的工具清单
├── mcp/{name}/{version}/        # MCP 工具，迁移自旧 {data_dir}/mcp/
└── plugin/{name}/{version}/     # Node.js 插件，迁移自旧 {data_dir}/plugin/
```

### 2.1 旧目录迁移

应用启动时自动迁移历史目录，无感知：

1. 若存在 `{data_dir}/mcp/`，整体移动到 `{data_dir}/plugins/mcp/`。
2. 若存在 `{data_dir}/plugin/`，整体移动到 `{data_dir}/plugins/plugin/`。
3. 更新 `config.json` 中所有 `ExtensionItem.RootDir` / `ConfigFilePath` 的前缀。
4. 迁移失败时不阻塞启动，记录日志并展示 banner 告警。

### 2.2 设计要点

* CLI 安装目录不带版本号 —— 同一个 CLI 同时只有一个版本，重装/升级直接覆盖。
* `cli_data/` 与 `cli/` 分离：删除插件只删 `cli/{name}/`，登录态等用户数据保留；用户可主动「重置数据」清 `cli_data/{name}/`。
* `cli_data/{name}/` 不随插件版本走，避免升级丢失登录态。

## 3. Manifest 设计

CLI 装好后，系统会引导 AI 阅读 `<cli> --help` 输出，生成下面这份 manifest，落盘到 `cli_data/{name}/manifest.json`。用户可在 UI 中编辑或重新生成。

```json
{
  "name": "lark-cli",
  "version": "1.2.3",
  "description": "飞书开放平台 CLI",
  "executable": "lark",
  "login_command": ["login"],
  "isolation": "isolated",
  "tools": [
    {
      "name": "lark_send_message",
      "description": "向指定用户/群发送消息",
      "input_schema": {
        "type": "object",
        "properties": {
          "to":      { "type": "string", "description": "接收方 open_id 或群 chat_id" },
          "content": { "type": "string", "description": "消息文本" }
        },
        "required": ["to", "content"]
      },
      "argv_template": ["message", "send", "--to", "{to}", "--content", "{content}"],
      "output_mode": "json",
      "timeout_seconds": 60,
      "requires_confirm": true,
      "enabled": true
    }
  ]
}
```

### 3.1 字段约定

* `executable`：CLI 入口的绝对路径或相对 `cli/{name}/node_modules/.bin/` 的命令名。安装时根据 `package.json` 的 `bin` 字段自动探测。
* `argv_template`：按字段名做位置替换（`{xxx}` → input.xxx）。**不经 shell 解释**，直接通过 `exec.Command(executable, argv...)` 调用，杜绝命令注入。
* `output_mode`：
  * `json`：先尝试 `JSON.Parse(stdout)`，成功放入 `parsed` 字段，失败回退到原文本。
  * `text`：原样返回。
  * `lines`：按 `\n` 切分成字符串数组。
* `requires_confirm`：AI 生成 manifest 时根据语义启发式判断（含 `send / delete / create / update / rm` 等动词默认 `true`，含 `list / get / show` 等默认 `false`），用户可在详情页改。
* `isolation`：
  * `isolated`（默认）：调用时注入 `HOME` / `USERPROFILE` / `XDG_*` 指向 `cli_data/{name}/` 下子目录。
  * `shared`：不注入，使用用户真实环境（适合外部已登录的场景）。

## 4. 安装流程

### 4.1 入口

提供三种入口，统一进入同一安装管线：

1. **设置页 → 插件与工具 → 添加 → CLI → 从 npm 安装**：弹输入框填 npm 包名。
2. **设置页 → 插件与工具 → 添加 → CLI → 从本地目录导入**：弹文件夹选择器，要求目录内包含 `package.json`。
3. **聊天页自然语言触发**：用户说"帮我装一个飞书 cli"，AI 调用内置 meta tool `install_cli_plugin(npm_package: string)` → 在聊天里弹审批气泡 → 用户确认 → 后台安装。

### 4.2 后端安装管线

```
1. 解析来源
   ├─ npm:   source = "@scope/pkg" 或 "pkg"
   └─ local: source = "/path/to/dir"

2. 校验插件运行时
   └─ 若 Node.js 运行时未就绪 → 报错并引导用户去 onboarding/runtime 设置页下载

3. npm install 到目标目录
   └─ npm install <source> --prefix {data_dir}/plugins/cli/{name} --no-audit --no-fund
      ⚠️ 全程使用内置 node/npm，env 中清除用户的 NPM_CONFIG_PREFIX 等可能影响隔离的变量

4. 解析包元信息
   ├─ 读 package.json 取 name / version / description / author / bin
   └─ executable 选择规则：
      ├─ bin 是字符串 → 用它
      ├─ bin 是对象  → 优先与包名同名的 entry
      └─ 都没有       → 报错"该 npm 包不是 CLI"

5. 探测 --help
   ├─ 在隔离 env 下运行 `<executable> --help`，超时 10s
   ├─ 失败回退尝试 `-h`、`help`
   └─ 若都失败 → 报错"无法获取 --help 输出，请检查该 CLI"

6. AI 生成 manifest 草稿
   ├─ 用当前默认聊天模型，prompt 输入 --help 文本 + 包元信息
   ├─ 输出符合 §3 schema 的 manifest JSON
   └─ 生成失败回退到空 tools 列表，由用户手动编辑

7. UI 展示草稿
   ├─ 用户可编辑工具列表、增删工具、调整 requires_confirm 等
   └─ 「保存并启用」 → manifest 落盘 → 注册到 agent 工具池

8. 持久化
   ├─ ExtensionItem 写入 config.json
   └─ manifest 写入 cli_data/{name}/manifest.json
```

### 4.3 进度反馈

* 安装期间通过 Wails 事件 `extension.cli.install.progress` 推送阶段：`installing / probing / generating_manifest / ready / failed`。
* 设置页与聊天页都监听该事件，分别在列表项上展示进度条 / 在聊天气泡内更新状态。

## 5. 调用流程

```
1. AI 决定调用 cli 工具 X，传入 input { to: ..., content: ... }

2. 工具调度层（agent/tools）
   ├─ 查 manifest 找到 X
   ├─ requires_confirm == true → 走现有审批面板
   └─ requires_confirm == false → 直接进入执行

3. 渲染命令
   ├─ argv = render(argv_template, input)        // 按字段名位置替换
   └─ 任何未在 input 中出现的占位符 → 报错给 AI（不要落到 shell）

4. 准备 env
   ├─ 基础：继承当前进程 env
   ├─ PATH 前置：cli/{name}/node_modules/.bin
   ├─ isolated 模式：覆盖 HOME / USERPROFILE / XDG_*
   └─ shared 模式：不覆盖

5. 执行
   ├─ exec.CommandContext(ctx, executable, argv...)
   ├─ 超时：manifest.timeout_seconds（默认 60s，硬上限 600s）
   └─ 截断：stdout / stderr 各自单边 32 KB 上限

6. 组装结果
   {
     "exit_code": 0,
     "stdout": "...",
     "stderr": "...",
     "parsed": { ... },          // 仅 output_mode=json 且解析成功时
     "truncated": false,
     "duration_ms": 123
   }

7. 返回 AI
   ├─ exit_code == 0 → 标记 success
   └─ exit_code != 0 → 标记 failure，stderr 一并回传，让 AI 决定下一步
```

## 6. 登录态管理

很多 CLI（飞书 / gh / aws）需要一次性 `login` 完成认证。提供两条登录路径，均使用同一段后端逻辑。

### 6.1 手动登录（详情页）

* 详情页提供「登录 / 认证」按钮，仅当 manifest 声明 `login_command` 时显示。
* 点击后后端跑 `<executable> <login_command...>`，stdout/stderr 实时回推到一个**前端登录弹窗**（可显示扫码 URL、提示文案、输入框）。
* 弹窗支持用户输入并通过 stdin 转发到子进程（覆盖飞书 / gh 这类需要回车确认的流程）。
* 同一时刻同一个 CLI 只允许一个登录会话。

### 6.2 AI 触发登录

* 系统在 manifest 注册时，若 `login_command` 非空，会额外为该插件注册一个工具 `{name}_login`。
* AI 在调业务工具拿到「未登录」类错误时，可调 `{name}_login` 工具，**总是要求用户审批**，审批通过后弹出与 §6.1 相同的登录弹窗。

### 6.3 隔离 vs 共享

* 详情页提供开关「使用系统全局登录态」：
  * 关（默认）：注入隔离 HOME / XDG 路径，token 存进 `cli_data/{name}/home/` 或对应 XDG 目录。
  * 开：不注入，使用用户真实 home。适合用户在系统外部已经登录过的场景。

### 6.4 重置数据

* 详情页提供「重置数据」按钮（红色二级按钮，二次确认）。
* 点击后清空 `cli_data/{name}/`，下次调用相当于全新安装的登录态。

## 8. 后端设计

### 8.1 新增包 `backend/pkg/cli/`（与 `pkg/mcp` 对位）

| 文件 | 职责 |
|---|---|
| `manager.go` | `Manager`：InstallFromNpm / InstallFromLocal / Uninstall / ResetData / Run / Login |
| `manifest.go` | manifest 类型定义、读写、AI 辅助生成（调用 agent 内部 LLM 接口） |
| `runner.go` | `exec.Command` 封装：env 隔离、超时、截断、output_mode 解析 |
| `npm.go` | 内置 npm 调用封装：使用 `service/runtime` 提供的 node/npm 路径，强制 `--prefix`，过滤危险 env |
| `probe.go` | `--help` 探测、`package.json` / `bin` 解析、executable 选定 |
| `*_test.go` | 单元测试 |

### 8.2 复用 `backend/service/plugin/`

* `ImportExtension` 输入扩展支持 `kind: "cli"`，并新增 `Source` 字段区分 `npm` / `local`。
* 列表 / 启用 / 禁用 / 删除流程复用，删除路由到 `cli.Manager.Uninstall`。
* 新增 Wails 方法：
  * `InstallCliFromNpm(input { package: string })`
  * `InstallCliFromLocal(input { path: string })`
  * `ResetCliData(input { id: string })`
  * `RegenerateCliManifest(input { id: string })`
  * `UpdateCliManifest(input { id: string, manifest: string })`
  * `LoginCli(input { id: string })` —— 启动登录会话
  * `SendLoginStdin(input { id: string, data: string })` —— 转发用户输入
* 新增 Wails 事件：
  * `extension.cli.install.progress`
  * `cli.login.output` / `cli.login.done`

### 8.3 启动迁移

在 storage / 服务初始化后、注册扩展之前调用 `MigrateLegacyExtensionDirs()`：

* 检测 `{data_dir}/mcp/` → 移动到 `{data_dir}/plugins/mcp/`
* 检测 `{data_dir}/plugin/` → 移动到 `{data_dir}/plugins/plugin/`
* 更新 `config.json` 中 `Extensions[*].RootDir` 与 `ConfigFilePath`
* 写入幂等标记（`{data_dir}/plugins/.migrated_v1`），避免重复执行

### 8.4 Agent 工具池

* CLI 插件启用后，按 manifest 把每个 `tool` 注册到 agent 工具池，工具 ID 命名规则：`cli_{name}_{tool_name}`（与现有 mcp 命名风格保持一致，需经过相同的 sanitize 处理）。
* 内置 meta tool `install_cli_plugin` 始终注册，不受插件启用状态影响。

## 10. 安全性

* **命令注入**：`argv_template` 严格按字段位置替换，不经 shell。任何包含未声明占位符的 manifest 拒绝保存。
* **npm 来源**：仅允许 `npm install <pkg>` 与本地路径，不允许 git url / tarball url（降低供应链风险）。后续可在「设置」中开放高级模式。
* **隔离 env**：默认 isolated 模式下，清除 `NPM_CONFIG_PREFIX`、`NODE_PATH` 等可能跨进程影响的变量；PATH 仅包含「插件运行时 bin + 当前 PATH」。
* **审批默认值**：AI 生成 manifest 时，写敏感词触发的工具默认 `requires_confirm: true`；meta tool `install_cli_plugin` 永远走聊天内审批气泡。
* **资源上限**：单次调用 stdout / stderr 各 32 KB 上限；超时默认 60s，硬上限 600s（manifest 无法超过）。

## 11. 验收要点

在干净环境下，按以下顺序应可全程走通：

1. 完成 onboarding，下载好插件运行时。
2. 在「插件与工具」点「添加 → CLI → 从 npm 安装」，输入一个公开 CLI 包名，能看到安装进度、manifest 草稿，确认后工具出现在 agent 工具池。
3. 在聊天里说「帮我装一个 XXX cli」，弹出审批气泡，确认后等同步骤 2。
4. 在聊天里让 AI 调用刚装好的 CLI 工具，命令执行成功，stdout 正确回传。
5. 调一个 `requires_confirm: true` 的工具，能看到审批面板，拒绝时 AI 收到拒绝提示。
6. 在详情页点登录，能完成一次交互式登录，且 token 落在 `cli_data/{name}/` 内。
7. 重启应用后旧 MCP 工具仍可用，且物理路径已迁移到 `{data_dir}/plugins/mcp/`。
8. 卸载 CLI 插件后 `cli/{name}/` 被清空但 `cli_data/{name}/` 仍在；点「重置数据」后 `cli_data/{name}/` 被清空。

## 12. 不在本期范围

* CLI 插件的版本升级 UI（本期靠重装覆盖）。
* CLI 插件的依赖共享（每个插件独立 `node_modules`，体积换隔离性）。
* CLI 插件作为「用户可直接使用的命令面板」入口（未来若需要再单独立项）。
* npm registry 镜像 / 私有 registry 配置（先用默认源，必要时手动配置 npm 环境变量）。
