# 面向普通用户的工具系统需求文档

本文是需求文档，只描述产品需求，不描述实现方案。

目标：

* 基于 `anomalyco/opencode` 的工具体系做调研
* 筛选适合普通用户产品的实用工具
* 明确工具权限系统需要满足的产品要求

> 实施时同时遵循 `docs/dev/00.rules.md`。

---

## 1. 调研结果

### 1.1 opencode 的工具特征

`opencode` 的工具体系最初是为 agent 执行复杂任务设计的，默认服务对象更接近开发者和高级用户，而不是普通消费用户。

它的工具大体可以分成几类：

* 系统执行类：
  * `bash`
* 文件与代码操作类：
  * `read`
  * `edit`
  * `write`
  * `apply_patch`
* 搜索与检索类：
  * `grep`
  * `glob`
  * `webfetch`
  * `websearch`
* 工作流类：
  * `question`
  * `skill`
  * `todowrite`
* 开发增强类：
  * `lsp`
  * `codesearch`
  * `task`
  * `plan`

### 1.2 对普通用户产品的启发

对于普通用户产品，不能把 opencode 的工具按原样暴露出来。

原因：

* 很多工具直接面向代码库、补丁、终端、LSP、子 agent 调度，这些都不是普通用户的自然任务表达。
* 普通用户关心的是“帮我完成任务”，而不是“让我选择和管理工具”。
* 如果工具暴露方式不对，产品会从“任务助手”变成“面向开发者的代理工作台”。

### 1.3 对本项目的结论

本项目是面向普通用户的，所以工具设计应该遵循以下原则：

* 工具以“任务能力”形式出现，而不是以开发工具名直接出现。
* 工具系统内部可以保留强能力，但外部必须收敛成普通用户可理解的动作。
* `bash` 必须保留，因为复杂任务离不开系统执行能力。
* `question`、`webfetch`、`websearch`、`skill` 是最值得优先参考的非代码工具。
* `todowrite` 可以作为内部流程工具保留，但不一定需要直接暴露给用户。
* `read`、`edit`、`write`、`apply_patch`、`grep`、`glob`、`lsp`、`task`、`plan` 这类工具不适合作为普通用户直面工具。

### 1.4 工具暴露方式结论

对普通用户产品，工具应该分成两层：

* 用户可感知层：
  * 网页搜索
  * 网页读取
  * 文件整理
  * 文件检索
  * 资料汇总
  * 调研报告生成
  * 澄清提问
* 系统内部执行层：
  * `bash`
  * `skill`
  * `todowrite`
  * 必要时受限的文件与搜索能力

需求重点不是“是否保留能力”，而是“如何把能力包装成普通用户能理解和信任的产品体验”。

## 2. 实用工具

本节只列适合本项目方向的实用工具需求。每个工具都要有清晰的工具描述、使用方式和用户故事。

### 2.1 Bash

工具描述：

* `bash` 是系统执行工具，用于在受控环境中执行 shell 命令。
* 对本项目来说，它不是给用户直接写命令的终端，而是内部承接复杂任务的底层能力。
* 它主要用来完成高层工具无法稳定覆盖的任务，例如文件整理、批量处理、目录扫描、外部程序调用、资料汇总等。

使用方式：

* 用户不直接看到 `bash` 这个工具名，也不直接输入 shell 命令。
* 用户通过自然语言发起任务，例如“帮我整理这个文件夹”“把这些资料整理成报告”。
* 系统在后台决定是否调用 `bash` 完成任务。
* 当任务涉及高风险动作时，系统必须在执行前向用户确认。

用户故事：

* 作为普通用户，我希望说“帮我把下载目录里的 PDF 按月份整理好”，系统能自动完成，而不是让我自己写命令。
* 作为普通用户，我希望说“帮我把这个文件夹里所有会议纪要汇总成调研报告”，系统能自动检索、整理并生成结果。
* 作为普通用户，我希望说“帮我找出这个目录里所有图片和文档，并列一个清单”，系统能直接给我结果。

### 2.2 Question

工具描述：

* `question` 是澄清与补充信息工具。
* 当用户需求不完整、存在多种可能方案、或者执行前需要确认偏好时，系统应主动提问。

使用方式：

* 以自然问答方式出现，不强调工具概念。
* 提问应简洁、明确，并尽量减少用户负担。
* 当系统无法安全执行或无法判断用户意图时，优先使用 `question` 而不是猜测。

用户故事：

* 作为普通用户，我希望当我说“帮我整理一下文件夹”时，系统会进一步问我要按时间、类型还是项目分类。
* 作为普通用户，我希望当系统准备执行可能影响很多文件的操作时，会先问我是否确认。
* 作为普通用户，我希望系统在信息不足时先问清楚，而不是直接做错。

### 2.3 WebFetch

工具描述：

* `webfetch` 是网页读取工具。
* 用于读取指定 URL 的页面内容，提取后续整理、总结、引用所需的信息。

使用方式：

* 用户通过粘贴链接或自然语言指定网页内容。
* 系统读取页面并把网页信息转成后续任务可用的数据。
* 该工具适合处理单个已知页面，而不是全网搜索。

用户故事：

* 作为普通用户，我希望贴一个活动页面链接后，系统能帮我提取关键信息。
* 作为普通用户，我希望贴一个帮助文档链接后，系统能帮我总结步骤。
* 作为普通用户，我希望贴一个商品或服务页面后，系统能帮我整理核心信息用于对比。

### 2.4 WebSearch

工具描述：

* `websearch` 是网页搜索工具。
* 用于搜索公开网络信息，补充本地资料和用户输入之外的外部信息。

使用方式：

* 用户通过自然语言提出搜索需求。
* 系统自动执行搜索、筛选和整理。
* 搜索结果应进一步加工为用户能直接使用的结论，而不是只返回原始链接列表。

用户故事：

* 作为普通用户，我希望说“帮我查一下这个主题的公开资料”，系统能直接帮我整理结果。
* 作为普通用户，我希望说“帮我搜一下最近的行业动态”，系统能给我摘要而不是一堆杂乱链接。
* 作为普通用户，我希望系统能把外部搜索结果和我本地资料结合起来生成报告。

### 2.5 Skill

工具描述：

* `skill` 是任务模板和流程说明工具。
* 它的作用是把某一类任务的固定处理流程沉淀下来，让系统在同类任务中表现更稳定。

使用方式：

* `skill` 更适合作为系统内部机制，而不是面向普通用户直接展示。
* 用户应看到的是“任务模板”“处理模式”或“预设流程”，而不是 `skill` 这个技术概念。
* 当某类任务反复出现时，系统应能复用稳定流程。

用户故事：

* 作为普通用户，我希望“调研报告生成”这种任务每次都按稳定格式输出，而不是每次风格差异很大。
* 作为普通用户，我希望“文件整理”这种任务在不同场景下也能保持一致的处理逻辑。
* 作为普通用户，我希望系统面对复杂任务时能像一个熟练助手一样按固定步骤做事。

### 2.6 TodoWrite

工具描述：

* `todowrite` 是任务拆解与进度跟踪工具。
* 它适合帮助系统在执行复杂任务时管理步骤和阶段。

使用方式：

* 普通用户未必需要直接看到 `todo` 工具。
* 更合适的方式是把它用作内部状态能力，必要时把进度转换成用户能理解的任务状态。
* 当任务较长、步骤较多时，可以向用户展示简化后的进度，而不是暴露原始工具形态。

用户故事：

* 作为普通用户，我希望在一个较长的整理任务中，能看到系统当前是在扫描文件、分类资料还是生成报告。
* 作为普通用户，我希望系统在执行多步骤任务时不会中途混乱或遗漏。
* 作为普通用户，我希望大型任务可以分阶段完成，并让我知道当前进展。

## 3. 权限系统

本项目保留强能力工具，尤其是 `bash`，因此权限系统必须成为核心需求，而不是可选补充。

### 3.1 权限系统目标

权限系统需要保证两件事同时成立：

* 系统有能力完成复杂任务。
* 用户始终对高风险操作有知情权和控制权。

### 3.2 权限系统的基本要求

权限系统至少需要支持以下三种动作：

* `allow`
  * 自动执行，不打断用户。
* `ask`
  * 执行前询问用户。
* `deny`
  * 不允许执行。

### 3.3 面向普通用户的权限原则

权限系统必须遵循以下原则：

* 普通用户不需要理解技术细节，但必须知道系统要做的事情是否会影响文件、目录或外部程序。
* 低风险动作尽量自动执行，减少频繁打断。
* 高风险动作必须显式确认。
* 高风险动作的确认文案必须是普通用户能理解的话，而不是技术术语。

### 3.4 需要重点控制的风险动作

以下动作必须作为重点权限场景：

* 批量移动文件
* 批量重命名文件
* 删除文件或覆盖文件
* 访问工作目录之外的路径
* 调用外部程序
* 长时间运行任务
* 输出过大或结果可能不完整的任务

### 3.5 对 Bash 的权限要求

因为 `bash` 是本项目必须保留的能力，所以必须单独提出要求：

* `bash` 只能作为内部能力使用，不应作为普通用户直接输入命令的界面能力。
* `bash` 必须有默认超时。
* `bash` 必须受工作目录限制。
* `bash` 访问目录外路径时，必须触发额外权限检查。
* `bash` 涉及批量文件修改、删除、移动时，必须在执行前询问用户。
* `bash` 的执行结果需要可解释，不能只返回原始命令输出。
* `bash` 的错误和超时结果需要转换成用户能理解的信息。

### 3.6 对普通用户确认体验的要求

确认交互必须满足以下需求：

* 用户要能看懂系统准备做什么。
* 用户要能知道这次操作会影响什么对象，例如哪些文件、哪个目录、哪类资料。
* 用户要能明确地选择“允许”“拒绝”。
* 对长任务或批处理任务，用户要能看到简洁的进度和结果说明。

### 3.7 权限系统与产品体验的关系

本项目不应把权限系统做成开发者式的复杂配置页为主，而应优先满足普通用户体验：

* 默认规则应尽量合理，减少用户手动配置负担。
* 重点是“关键时刻 ask”，不是让用户从一开始配置一大堆技术选项。
* 权限系统要服务于信任感，让用户知道系统既能做事，也不会乱做事。

### 3.8 本期需求结论

本期工具系统需求应明确为：

* 保留 `bash`，但把它作为内部能力封装使用。
* 优先建设 `question`、`webfetch`、`websearch`、`skill` 这几类更贴近普通用户任务表达的工具能力。
* `todowrite` 作为复杂任务内部状态能力保留。
* 建立一套面向普通用户的权限系统，用来约束 `bash` 和其他高风险工具动作。
