# CLI 智能安装升级需求

本文定义 CLI 智能安装能力的下一阶段需求，重点解决“只完成安装，不继续初始化”的体验断点。本文以 `lark-cli` 为一期主场景，同时要求本期落地通用的“CLI 初始化编排”能力，为 `aws-cli`、`gcloud` 等不同认证方式提供统一承载。该能力的核心原则不是把流程写死在产品逻辑里，而是由 AI 基于 skill 指引、自主判断安装后的下一步动作。

> 实施时同时遵循 `docs/dev/00.rules.md`。

---

## 1. 背景

当前 CLI 安装链路已经具备“装上工具并生成 manifest”的能力，但还不具备“自动把工具初始化到可用状态”的能力。

结合现有代码，当前状态大致如下：

* `backend/pkg/skills/_builtin/install-cli-from-docs/SKILL.md`
  * 内置 skill 目前只覆盖：识别包名 → `InstallCli` → `GenerateCliManifest`。
  * 文档中明确只要求“提示用户后续登录”，没有要求自动继续执行初始化。
* `backend/pkg/agent/tools/install_cli_tool.go`
  * 安装工具只有 `InstallCli`，语义仍是“安装一个 npm CLI”。
* `backend/service/plugin/plugin_cli.go`
  * 已有 CLI 安装、manifest 生成、配置持久化能力。
* `backend/service/plugin/plugin_login.go`
  * 已支持按 manifest 中的 `login_steps/login_command` 启动交互式登录流程，并通过事件把 PTY 输出推到前端。
* `frontend/src/components/settings/plugins/CliLoginDialog.tsx`
  * 已有通用 PTY 登录弹窗，适合“人工盯着终端完成登录”，但不适合“后台安装任务自动进入认证编排，再把必要信息结构化展示给用户”。

因此，当前“智能安装”本质上仍是“智能安装 + 手动初始化”，用户体验是不连续的。更关键的是，当前链路的能力边界仍偏向固定动作调用，而不是“让 AI 在技能提示下自主完成一整条初始化任务”。

---

## 2. 当前问题

当用户说“帮我安装飞书 CLI，文档在 ……”时，系统只完成：

1. 下载安装包
2. 生成 manifest

但不会继续：

1. 检查是否需要初始化
2. 自动触发登录 / 配置命令
3. 识别认证交互模式
4. 向用户展示授权入口与二维码
5. 在用户授权后续上同一轮初始化，直至验证完成

这会带来几个明显问题：

* 用户心智断裂：用户以为“安装”意味着“可以用了”，实际上只是“文件装好了”。
* 对话不连续：模型没有把安装、配置、登录、验证视为同一条任务链。
* 认证体验差：像 `lark-cli` 这类设备码登录流程，如果直接跑阻塞命令，用户既看不到清晰的引导，也难以在正确时机完成授权。
* 无状态续接：当认证需要跨轮等待用户授权时，当前链路没有一个面向“安装任务”的临时状态承载。

---

## 3. 目标

本期目标不是把所有 CLI 的初始化细节都做成同样深度的自动化，而是把“安装后继续初始化”升级成一项通用平台能力，并先用 `lark-cli` 做主验收场景。

一期目标：

* 用户通过自然语言或“从官方文档安装”入口接入 `lark-cli` 时，系统在安装完成后自动进入初始化流程，而不是停在安装成功。
* 本期需要落地通用的“CLI 初始化编排”能力，将安装、配置、登录、验证视为同一条连续任务链，而不是为 `lark-cli` 单独写死特例。
* 初始化过程应尽量由 AI 自主决策：skill 负责提供流程提示、约束和注意事项，AI 负责结合当前环境、CLI 输出和已有状态决定下一步动作。
* 当初始化进入设备授权阶段时，系统能自动切换到适合的交互策略：
  * 先拿到 `verification_url` 与 `device_code`
  * 向用户展示原始授权链接
  * 自动生成并展示二维码图片
  * 在用户完成授权后续上同一个 `device_code` 完成登录
* 整个过程对用户尽量透明，除“去浏览器完成授权”外，不要求用户再手工输入命令。
* 本期的通用编排能力要能承载至少三类初始化模式：
  * 设备码登录
  * 浏览器跳转登录
  * API Key / 配置型初始化

在本期中，`lark-cli` 是主验收对象；`aws-cli`、`gcloud` 等 CLI 则用于约束方案不能写死为飞书特例。

---

## 4. 一期范围

### 4.1 主场景

以 `lark-cli` 为一期唯一强保障场景，覆盖以下完整路径：

1. 安装 `lark-cli`
2. 生成 / 修正 manifest
3. 检查当前是否已登录
4. 若未登录，则以设备码模式发起认证
5. 展示原文授权 URL 与二维码
6. 等待用户授权
7. 使用首次拿到的 `device_code` 完成轮询
8. 验证登录成功
9. 向用户反馈“CLI 已可用”

### 4.2 通用能力范围

除 `lark-cli` 主场景外，本期还需要实现通用能力边界：

* 初始化流程不再等同于“执行一个 login_command”，而是支持多阶段编排。
* 编排框架要能区分至少以下类型：
  * 设备码登录
  * 浏览器授权登录
  * API Key / Token 输入
  * 配置文件 / 环境检查 / 验证步骤
* 对 `aws-cli`、`gcloud` 等 CLI，本期不要求把所有供应商特定分支全部打磨完，但要求通用框架已经可以承载这些流程，而不是需要另起一套产品模型。

## 5. 核心体验要求

### 5.0 通用原则

本期不应将“初始化编排”设计成高度固定的产品流程模板，例如把某个 CLI 永久绑定为：

1. 安装
2. 运行固定登录命令
3. 等待用户授权
4. 验证

更合适的原则是：

* 产品只提供通用能力边界：
  * 安装任务承载
  * 跨轮状态保存
  * 用户介入与通知
  * 结构化展示能力
  * 二维码生成能力
* skill 提供针对某类 CLI 或认证模式的操作提示。
* AI 根据文档、环境探测结果、CLI 返回内容和历史状态，自主决定下一步执行什么。

也就是说，本期要做的是“让 AI 能完成初始化编排”，而不是“把初始化编排硬编码成固定状态机”。

### 5.1 安装不再等于结束

无论是聊天里的“帮我安装某 CLI”，还是设置页“从官方文档安装”，只要系统识别该 CLI 在首次可用前还需要额外初始化步骤，就不能在“安装完成”时结束任务。

正确的结束条件应改为：

* 已安装
* 已完成必需初始化
* 已完成可用性验证

只有三者都满足，任务才算真正完成。

### 5.2 初始化过程连续

安装任务在后台必须被视为一条连续流程，而不是三个互相脱节的动作：

1. 安装
2. 初始化 / 登录
3. 验证

用户看到的是“同一个任务正在继续推进”，而不是“安装完了，你再自己去点登录”。

### 5.3 用户只在必要处介入

对 `lark-cli` 而言，用户唯一需要做的事情是：

* 点击或扫码授权链接
* 在浏览器内确认授权

除此之外，不应要求用户再回到终端输入命令，也不应要求用户手工复制 `device_code`。

## 6. lark-cli 一期标准流程

### 6.1 阶段一：安装与基本探测

系统完成现有能力：

* 根据文档识别 `lark-cli`
* 安装 CLI
* 生成 manifest

在此基础上新增一步：

* 自动判断该 CLI 是否存在“首次使用前必须初始化”的要求

对 `lark-cli`，当前应高概率判断为“需要继续执行认证初始化”；但从产品要求上，这个结论应由 AI 基于 skill 和探测结果得出，而不是只靠前端或后端硬编码分支决定。

### 6.2 阶段二：获取认证信息

系统不得直接运行阻塞式 `lark-cli auth login` 再把终端输出原样丢给用户，而应采用适合设备码登录的方式。

对 `lark-cli`，标准行为为：

* 优先执行 `lark-cli auth login --no-wait --json`
* 从结果中提取：
  * `verification_url`
  * `device_code`
  * 过期时间或有效期（若返回中可得）

此阶段的产出是结构化认证信息，而不是终端文本流。这里的关键不是把 `--no-wait --json` 固化成某个后端专用逻辑，而是要允许 skill 明确提示 AI：“在设备码登录场景下，应先获取结构化认证信息，再决定如何向用户展示”。

### 6.3 阶段三：展示认证入口

在拿到认证信息后，系统必须按以下顺序向用户展示：

1. 原文 `verification_url`
2. 该 URL 对应的二维码图片
3. 简短说明：该链接将在有限时间内有效，请尽快完成授权

要求如下：

* URL 必须原样展示，不得改写。
* 二维码优先用 PNG 生成。
* 仅当用户明确要求 ASCII 形式时，才退化成终端字符二维码。
* 仅生成二维码文件不算完成，必须在回复里实际展示出来。

### 6.4 阶段四：等待用户授权

当认证入口展示完成后，本轮任务应停在“等待用户授权”状态，而不是继续立即发起阻塞式轮询。

这是因为：

* 若先展示 URL，再立刻阻塞执行轮询，容易造成时序错误或超时浪费。
* 设备码必须复用首次申请得到的 `device_code`，不能在等待后重新发起一轮新的认证。

因此，该阶段需要显式保留认证上下文，等待用户反馈“已完成授权”或系统检测到授权完成后，再继续下一步。

### 6.5 阶段五：续接登录并验证

在用户完成授权后，系统继续使用首次获取的 `device_code` 续接流程，执行等价于：

* `lark-cli auth login --device-code <code>`

然后补一层验证：

* 验证登录成功
* 若需要，再执行轻量只读命令确认 token 可用

任务成功的最终反馈应是“已安装并已完成认证，可继续使用”，而不是只有“登录命令退出码为 0”。

## 7. 产品能力要求

### 7.1 流程动态识别

系统需要能够判断“安装后是否还有未完成初始化动作”，而不是永远固定为：

* 安装成功 = 任务结束

一期可先对 `lark-cli` 采用规则化策略，但对外的产品定义必须是通用的：

* 安装后检查环境
* 判断是否需要登录 / 配置 / 验证
* 再决定是否继续下一阶段

这里强调的是“AI 决定流程”，而不是“产品预先把所有路径写死”。产品负责提供状态容器与交互能力，具体走哪条路径由 AI 结合 skill 决定。

### 7.2 认证交互自适应

不同 CLI 的认证方式不同，产品上要承认这种差异，并要求系统采用匹配的用户交互：

* 设备码登录：展示链接 + 二维码 + 跨轮续接
* 浏览器跳转登录：展示链接并等待完成
* API Key / Token：向用户索取并保存
* 本地配置文件：自动生成或引导确认

本期要求把以上差异纳入同一套初始化编排能力中，而不是仅在需求层面预留。`lark-cli` 仍是主验收对象，但产品结构、状态定义和交互设计必须能直接覆盖 `aws-cli`、`gcloud` 等场景。

### 7.3 初始化编排能力

本期需要把“CLI 初始化编排”定义为正式能力，而不是零散规则集合。其核心要求包括：

* 将安装后的动作组织成可连续推进的任务链，例如：
  * 环境检查
  * 配置生成
  * 登录发起
  * 用户授权等待
  * 登录续接
  * 可用性验证
* 任务链不要求预先静态穷举；更合理的方式是由 AI 在 skill 提示下动态决定当前步骤和下一步。
* 每个阶段都要有明确状态，允许成功、失败、等待用户、超时、重试。
* 编排过程要能被聊天场景和设置页场景复用，而不是分别实现两套逻辑。
* 编排能力本质上应是“AI 驱动的任务执行容器”，而不是“产品内置的固定流程引擎”。

### 7.4 Skill 驱动

本期需要明确 skill 在初始化编排中的角色：

* skill 负责告诉 AI：
  * 某类 CLI 常见有哪些初始化步骤
  * 遇到什么输出时应如何判断
  * 哪些命令适合先探测、哪些命令适合阻塞执行
  * 哪些场景必须展示二维码、哪些场景要索取 token
  * 哪些临时状态必须跨轮保存
* AI 负责根据 skill 指引和实际运行结果，自主选择：
  * 现在要不要继续
  * 应该执行哪条命令
  * 是否需要用户介入
  * 要向用户展示什么信息

产品侧不应假设所有 CLI 都遵循同一个固定脚本。对同一类 CLI，skill 也应允许持续演进，而不要求每次新增适配都改宿主产品逻辑。

### 7.5 状态持久化

当初始化不是一轮内可闭合时，系统必须为任务保存临时状态，至少包括：

* 当前阶段
* 目标 CLI
* `device_code`
* `verification_url`
* 失效时间
* 最近一次错误

这样后续用户回复“我已经授权好了”时，系统才能续接上一轮，而不是重新开始一轮新登录。

### 7.6 超时与重试

如果用户在有效期内未完成授权，系统不应卡死或静默失败，而应：

* 明确提示授权已超时
* 提供“重新发起认证”能力
* 丢弃旧 `device_code`
* 生成新的认证信息并重新展示

## 8. 界面与交互要求

### 8.1 聊天场景

当用户在聊天中发起“安装 `lark-cli`”时：

* 任务进度继续沿用现有后台安装思路
* 当进入设备授权阶段时，聊天回复中要出现：
  * 可点击的原始 URL
  * 二维码图片
  * 一句简短说明

当用户完成授权后回复“已授权”：

* 系统应自动续接此前任务
* 成功后给出明确成功提示

### 8.2 设置页场景

当用户通过“从官方文档安装”入口接入 `lark-cli` 时：

* 列表项不能只停在“已安装”
* 需要有可区分的中间状态，例如：
  * 安装中
  * 初始化中
  * 等待授权
  * 验证中
  * 已就绪

若授权必须由用户在聊天里完成，也应给出清晰跳转或通知，不应让列表状态无声卡住。

### 8.3 二维码展示

二维码是本期 `lark-cli` 场景的必选项，不是锦上添花功能。

验收标准上必须满足：

* 二维码由系统生成
* 二维码以图片形式展示给用户
* 展示顺序为“URL 在上，二维码在下”

## 9. 对现有能力的改造要求

本文只定义需求，不展开具体技术设计；但为了便于研发理解，这里明确几个必须变化的现状。

### 9.1 内置安装 skill 需要升级

当前 `install-cli-from-docs` skill 只做到“安装 + 生成 manifest + 提示可能需要登录”。

本期要求升级为：

* 安装后继续判断初始化需求
* 进入通用的 CLI 初始化编排，而不是只做一次性安装
* 对 `lark-cli` 给出清晰的设备码登录指引
* 对其他 CLI 允许通过不同 skill 或同一 skill 的不同提示段，进入浏览器授权、API Key、配置型初始化分支
* 支持在等待授权时保留上下文，并在后续回合续接

更理想的形态是：

* “从官方文档安装 CLI”不只是一个安装 skill
* 它同时也是“安装后初始化编排”的主入口 skill
* 后续可按认证模式拆出子 skill，或通过同一 skill 内的模式提示让 AI 自主选择

### 9.2 登录能力不能只等同于 PTY 弹窗

当前 `plugin_login.go + CliLoginDialog.tsx` 更适合“人工接管式登录”。

本期新增需求后，系统还需要一种更高层的能力：

* 能把认证信息结构化提取出来
* 能区分“现在该展示链接”与“现在该继续轮询”
* 能在非 PTY 直通模式下完成设备码登录编排
* 能承载不依赖 PTY 的初始化步骤，如浏览器授权提示、token 收集、配置检查与验证

PTY 弹窗能力仍然保留，但不应再是 `lark-cli` 智能安装场景的唯一承载方式。

### 9.3 安装任务需要有初始化状态

当前安装任务更接近“下载与接入任务”。

本期之后，安装任务的产品语义应升级为：

* CLI 接入与首次初始化任务

因此任务状态、列表文案、成功定义都需要随之调整。

## 10. 产品边界 vs AI 边界

为了避免实现过程中重新滑回“把每个 CLI 的初始化流程都硬编码进宿主”，本期需要明确区分哪些能力必须由产品提供，哪些能力应该交给 AI 在 skill 提示下自主决策。

### 10.1 产品边界

以下能力属于宿主产品必须稳定提供的通用基础设施：

* CLI 安装与执行能力
  * 安装 npm CLI
  * 执行命令
  * 收集 stdout / stderr / exit code
  * 支持超时、取消、重试
* 任务承载能力
  * 能创建“安装 + 初始化”任务
  * 能在后台持续运行
  * 能表达运行中、等待用户、失败、完成等状态
* 状态持久化能力
  * 能跨轮保存任务上下文
  * 能保存临时认证信息和最近一次错误
* 用户交互能力
  * 能向用户展示链接、文本、结构化提示
  * 能展示二维码图片
  * 能在必要时向用户索取输入
  * 能发通知、恢复任务、继续任务
* 安全与约束能力
  * 能限制命令执行边界
  * 能控制超时与重试
  * 能记录关键操作和任务状态
  * 能承载“是否需要用户确认”的审核机制，并允许 AI 给出动态确认决策

产品边界的核心原则是：

* 只提供通用执行与交互能力
* 不把某个具体 CLI 的初始化流程写死在宿主代码里

### 10.2 AI 边界

以下能力应交给 AI 结合 skill、自主判断：

* 安装后是否还需要继续初始化
* 当前最合理的下一步动作是什么
* 某条输出意味着“已登录 / 未登录 / 需要授权 / 需要补配置 / 需要重试”
* 应优先用哪种认证策略
  * 设备码
  * 浏览器跳转
  * API Key
  * 配置文件初始化
* 是否需要请求用户介入，以及该向用户展示什么
* 当前这次工具调用是否应该要求用户确认
* 何时可以宣布“CLI 已真正可用”

AI 边界的核心原则是：

* AI 负责理解上下文与决策路径
* skill 负责给 AI 提供模式提示、风险提示和推荐动作
* 宿主不替 AI 预先穷举所有 CLI 分支

### 10.3 Skill 边界

skill 处于产品与 AI 之间，承担“流程知识注入”的职责。

skill 主要负责：

* 描述某类 CLI 常见初始化模式
* 指出推荐探测命令与危险命令
* 告诉 AI 哪些输出值得关注
* 告诉 AI 哪些信息必须展示给用户
* 告诉 AI 哪些状态必须跨轮保存
* 告诉 AI 哪些类型的工具调用通常应要求用户确认，哪些可以默认直跑

skill 不应负责：

* 直接替代宿主产品的状态管理
* 假设某个具体 UI 控件一定存在
* 绕开宿主提供的安全边界

### 10.4 边界划分示例

以 `lark-cli` 为例：

* 产品负责：
  * 运行 `lark-cli auth login --no-wait --json`
  * 保存 `device_code`
  * 展示授权 URL 与二维码
  * 在用户回复后恢复任务
* skill 负责：
  * 提示 AI 应优先使用 `--no-wait --json`
  * 提示 AI 不要重新申请新的 `device_code`
  * 提示 AI 展示顺序必须是 URL 在上、二维码在下
* AI 负责：
  * 判断当前是否已经需要进入认证
  * 判断何时向用户展示授权信息
  * 判断用户回复后是否该继续轮询
  * 判断某一步是否需要先征求用户确认
  * 判断何时可以宣布登录成功

以 `aws-cli` 或 `gcloud` 为例：

* 产品仍然负责相同的底层能力：
  * 执行命令
  * 保存状态
  * 请求用户输入
  * 展示链接或二维码
* skill 改变的是流程知识：
  * 可能改成浏览器授权
  * 可能改成 API Key 输入
  * 可能改成配置文件初始化
* AI 改变的是决策路径：
  * 根据实际输出选择下一步，而不是走一套固定飞书流程

### 10.5 工具审核逻辑改造要求

本期还需要同步改造工具执行的审核逻辑。

当前方向不应再是：

* 仅依赖工具静态配置中的固定字段决定是否确认
* 仅按工具名或 CLI 名字硬编码“这个一定确认 / 那个一定不确认”

更合适的目标是：

* 由 AI 基于当前上下文动态判断这一次调用是否需要用户确认
* skill 为 AI 提供判断提示
* 产品提供最终的审核承载与安全兜底

也就是说，审核决策应从“静态工具属性”升级为“AI 主导的运行时决策”。

### 10.6 审核决策分工

在“是否需要用户确认”这件事上，三层分工应如下：

* 产品负责：
  * 提供统一确认弹层 / 审批交互
  * 记录本次调用是否经过确认
  * 为高风险操作保留不可绕过的安全兜底
* skill 负责：
  * 告诉 AI 哪类动作通常是高风险
  * 提示 AI 哪类只读动作通常可以直接执行
  * 提示 AI 在哪些不确定场景下应倾向于先确认
* AI 负责：
  * 结合当前参数、命令意图、目标对象和上下文，判断这一次调用要不要确认

例如：

* 同一个 CLI 工具，在只读查询场景下可以免确认
* 但在写入、删除、发送消息、修改配置、上传文件等场景下，AI 应主动要求确认
* 即使是同一个命令模板，也不能只靠“工具名固定标签”做永久判断

### 10.7 审核逻辑的兜底原则

虽然本期要求让 AI 决定是否需要确认，但产品侧仍需保留底线约束，避免完全失控。

至少应满足：

* 明显高风险动作必须允许产品侧强制要求确认
* 当 AI 无法判断风险时，应默认倾向于确认，而不是静默直跑
* 审核系统需要能区分：
  * AI 主动要求确认
  * 产品规则强制确认
  * AI 判定可直接执行
* 后续审计或问题排查时，应能看出一次调用为何被要求确认，或为何被直接执行

这意味着本期的审核改造目标不是“把所有决定都交给 AI 后不设限制”，而是：

* AI 负责更细粒度、更贴近语义的确认判断
* 产品负责统一承载、底线兜底和可追踪性

### 10.8 设计约束

后续实现若出现以下倾向，应视为偏离本期目标：

* 在宿主代码中新增大量 `if cli == "lark-cli"` 形式的特判流程
* 保留或继续扩展此前为了适配 `lark-cli` 而加入的固定流程分支，而不回收到通用编排能力中
* 把认证流程顺序硬编码为固定状态机，导致 skill 只能填空不能决策
* 审核是否确认仍完全由静态 `requires_confirm` 一类字段决定，AI 只能被动接受
* 把跨轮状态全部塞进 AI 文本上下文，而不是宿主可恢复状态
* 把二维码、链接展示、用户输入采集等通用能力做成某个单一 CLI 特供功能

正确方向应是：

* 宿主负责稳定通用能力
* skill 负责持续演进知识
* AI 负责在运行时完成判断和编排

### 10.9 既有飞书特例清理要求

本期实现通用初始化编排时，需要主动清理此前为了适配 `lark-cli` 而形成的固定流程思路，避免新能力只是“在旧特例上再包一层”。

具体要求：

* 不能以“先把 `lark-cli` 特例保留，其他 CLI 再慢慢接入”为默认实现路径。
* 现有与 `lark-cli` 强绑定的流程判断、状态命名、交互分支，应尽量回收到通用初始化编排抽象中。
* 现有与某个 CLI 强绑定的审核确认逻辑，也应同步收敛到通用审核能力中。
* `lark-cli` 可以继续作为主验收样例，但不应继续拥有宿主层面的长期专属流程。
* 若短期内仍需保留少量兼容分支，必须以“过渡实现”对待，并以收敛到通用能力为明确目标，而不是形成新的永久层。

### 10.10 内置 Node.js 运行时要求

CLI 安装与初始化过程中，凡涉及 `npm`、`npx`、Node.js 执行环境的场景，都必须使用应用内置的 Node.js 运行时，不得依赖用户系统全局安装的 Node.js。

明确要求如下：

* 安装 CLI 时，`npm` 必须来自应用内置运行时。
* 解析、探测、生成 manifest 时，相关 Node.js 执行链路必须基于内置运行时。
* 初始化过程中如果需要再次调用 `npm` / `npx` / Node.js 脚本，也必须继续沿用内置运行时。
* 不得因为用户本机存在全局 `node` / `npm`，就回退到系统环境执行。

这样做的目的包括：

* 保证安装与初始化环境一致
* 降低用户本机环境差异带来的不确定性
* 避免因 PATH、版本漂移或全局 npm 配置导致行为不一致
* 维持 CLI 插件体系的可控性与可复现性

## 11. 非目标

本期不要求：

* 一次性打磨完所有 CLI 的供应商细节
* 自动替用户完成浏览器内授权点击
* 设计一个通用的凭据保险箱产品
* 替换所有现有手动登录入口
* 完整重构 manifest 生成逻辑

本期的成功标准是：

* 通用“CLI 初始化编排”能力落地
* `lark-cli` 主链路完整跑通
* 同一套产品模型可直接承载 `aws-cli`、`gcloud` 等后续适配
* 初始化过程主要由 AI 基于 skill 自主决策，而不是依赖大量 CLI 特例硬编码

## 12. 验收标准

以下场景全部通过，才算本期需求达成：

1. 用户输入“帮我安装飞书 CLI，文档在 <URL>”，系统在安装完成后不会直接结束，而会自动进入认证初始化流程。
2. 当进入认证阶段时，回复中包含：
   * 原文授权 URL
   * URL 下方的二维码图片
   * 有效期提示或“请尽快完成授权”的提示
3. 系统不会在展示 URL 后立刻重新发起一轮新的认证，也不会丢失首次生成的 `device_code`。
4. 用户完成授权并回复后，系统能够续接此前上下文，完成登录轮询与成功验证。
5. 整个过程中，用户不需要手动输入任何 CLI 命令。
6. 若用户超时未授权，系统能明确提示并支持重新发起认证，而不是卡死在等待状态。
7. 设置页中的 CLI 安装状态能够区分“已安装但未完成初始化”和“已完全可用”。
8. 本期产出的初始化编排能力可以同时表达以下差异化流程，而不需要再新建独立产品方案：
   * 设备码登录
   * 浏览器授权登录
   * API Key / Token 输入
   * 配置检查与验证步骤
9. `lark-cli` 作为样例跑通时，其成功不依赖硬编码固定流程，而是依赖 AI 在 skill 提示下完成探测、判断、展示、续接与验证。

## 13. 后续扩展方向

在 `lark-cli` 跑通且初始化编排能力落地后，后续可沿同一产品模型继续扩展更多 CLI：

* `aws-cli`
  * 典型是 Access Key / SSO / 浏览器认证混合模式
* `gcloud`
  * 常见为浏览器授权与本地配置写入
* `gh`
  * 常见为设备码 / 浏览器授权

扩展时建议沿用同一套产品概念：

* 安装任务不在“安装结束”即完成
* 初始化被视为任务的一部分
* 认证方式按 CLI 类型自适应
* 临时状态支持跨轮续接
* skill 持续演进，AI 根据 skill 自主调整流程

这样用户最终感知到的会是统一体验：

* “帮我装好并准备到可用状态”

而不是：

* “帮我把文件放到机器上，剩下你自己处理”
