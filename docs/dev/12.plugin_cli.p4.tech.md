# CLI 插件实施计划 — P4 AI 生成 manifest + 工具池注册

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让 CLI 插件接入 AI：(a) 安装后由当前默认聊天模型读 `--help` 自动生成 manifest 草稿；(b) 启用后的 CLI 工具注册到 agent 工具池，AI 在聊天里可直接调用，`requires_confirm: true` 的工具走现有审批面板。

**Architecture:** 后端新增轻量级一次性 LLM 调用 helper（绕过流式 agent runner）用于 manifest 生成；扩展 `agent.Manager.buildCliToolSets` 与 `buildMCPToolSets` 对称，把 cli ExtensionItem 的 manifest.tools 包装成 `agenttool.ToolSet`，单次工具调用代理到 `cli.Manager.RunTool`。前端在详情页加「重新生成 manifest」按钮与工具列表，按需触发自动生成。

**Tech Stack:** Go + trpc-agent-go + Wails v3 + React + Tailwind。

参考需求文档：`docs/dev/12.plugin_cli.md` §3 Manifest、§4.2 步骤 6（AI 生成）、§5 调用流程、§7 详情面板工具列表。所有实现遵循 `docs/dev/00.rules.md`。

## 范围与不在范围

**本期实现：**
- AI 读 `--help` + 包元信息，生成符合 §3 schema 的 manifest 草稿（失败回退到空 tools）。
- 「重新生成 manifest」UI 入口（详情页按钮，带二次确认）。
- 启用 CLI 插件后，每个 `enabled` 的 manifest tool 自动出现在 agent 可用工具池中，AI 可直接调用。
- 工具调用结果（含 truncated / parsed）回传给 AI，`requires_confirm: true` 的调用走现有审批面板。

**不在本期：**
- 安装时自动触发生成（本期是手动按钮；后续可改成安装完成后默认弹窗确认是否生成）。
- 工具入参 JSON Schema 高级特性（嵌套对象、union、enum 强制校验等）—— 本期只做基础 string/number/bool 字段传递。
- 登录流程（P5）。
- 聊天端 meta tool `install_cli_plugin`（P6）。
- Windows 端 CLI 工具调用（继承 P2 的 Unix-only 实现）。

```go
type OneshotRequest struct {
	BaseURL      string
	APIKey       string
	ModelName    string
	ProviderType pkgProvider.Type
	System       string
	User         string
	MaxTokens    int           // 0 → 由 SDK 默认
```

## Task 2: ManifestGenerator — 把 --help 输出转成 Manifest

**Files:**
- Create: `backend/pkg/cli/generator.go`
- Create: `backend/pkg/cli/generator_test.go`

### 接口

```go
// GenerateParams 是生成 manifest 草稿所需输入。
type GenerateParams struct {
	HelpText    string         // <executable> --help 的 stdout
	PackageName string         // 来自 package.json
	PackageMeta PackageMeta    // 完整 package.json 元信息（用于 prompt 上下文）
	Executable  string         // 已选定的可执行入口（落到 manifest.Executable）
	Caller      OneshotCaller  // 注入：调用 LLM 的回调，便于单测注入 mock
}

// OneshotCaller 是 generator 对外的依赖注入点。实现可以直接是 agent.OneshotComplete 的闭包。
type OneshotCaller func(ctx context.Context, system, user string) (string, error)

// Generate 调 LLM 生成 manifest 草稿，校验通过后返回；失败回退到 zero tools 的 stub manifest。
func Generate(ctx context.Context, p GenerateParams) (Manifest, error)
```

### Prompt 设计（写在 generator.go 内为常量）

```
SYSTEM:
You convert a CLI tool's --help output into a JSON manifest for the Lemontea CLI plugin system.

The manifest schema is:
{
  "name": "<package name>",
  "version": "<package version>",
  "description": "<short>",
  "executable": "<provided>",
  "login_command": ["<optional, the login subcommand if the CLI requires authentication>"],
  "isolation": "isolated",
  "tools": [
    {
      "name": "<snake_case_tool_name>",
      "description": "<what it does>",
      "input_schema": { "type": "object", "properties": { "<key>": {"type":"string"} }, "required": [] },
      "argv_template": ["<subcmd>", "--flag", "{key}"],
      "output_mode": "json" | "text" | "lines",
      "timeout_seconds": 60,
      "requires_confirm": true|false,
      "enabled": true
    }
  ]
}

Rules:
- Map each useful subcommand to a tool. Skip "help" / "version" / "completion".
- requires_confirm: true for verbs that mutate state (send/delete/create/update/upload). false for read verbs (list/get/show).
- argv_template must use {placeholder} tokens that match input_schema.properties keys.
- output_mode: prefer "json" if the CLI has a --json flag for that subcommand, else "text".
- Return ONLY the JSON object, no prose.

USER:
Package: {{PackageName}} v{{PackageMeta.Version}}
Description: {{PackageMeta.Description}}
Executable: {{Executable}}

--help output:
```
{{HelpText}}
```
```

### 实现要点

1. 渲染 prompt 后调 `Caller(ctx, system, user)`，拿到 raw 字符串。
2. 容错解析：模型可能在 JSON 前后加 ` ```json ... ``` ` 或解释文字。先尝试 `json.Unmarshal`；失败时截取第一个 `{` 到最后一个 `}` 之间的子串再试。
3. 把 `Executable` 字段强制覆盖为调用方传入的值（防止模型自己幻觉一个路径）。
4. 调 `Validate(manifest)`，失败时回退到「空 tools 的最小 manifest」并返回 nil error（用户可手工编辑），但 log warning。

### 测试

- Mock `Caller` 直接返回提前准备好的 JSON 字符串：验证字段解析正确、Executable 被覆盖、tools 数量符合预期。
- Mock `Caller` 返回 ` ```json {…} ``` `：验证 fence 容错。
- Mock `Caller` 返回乱码：验证回退到空 tools 的 stub manifest（不报错）。
- 单元层面**不**测试真实 LLM；那个由 Task 1 的 mock-server 测试已覆盖。


### DTO

```go
// GenerateCliManifestInput identifies which CLI plugin to regenerate the manifest for.
type GenerateCliManifestInput struct {
	ID string `json:"id"`
}

// GenerateCliManifestOutput returns the refreshed extension after a successful generation.
type GenerateCliManifestOutput struct {
	Extension data_models.ExtensionItem `json:"extension"`
}
```

### Wails 方法

```go
// GenerateCliManifest probes the CLI's --help output, asks the current default chat model
// to draft a manifest, validates it, and persists the result. Falls back to an empty-tools
// manifest if the model output cannot be parsed/validated.
func (p *Plugin) GenerateCliManifest(ctx context.Context, input GenerateCliManifestInput) (*GenerateCliManifestOutput, error)
```

实现步骤：

1. 从 config.json 取 ExtensionItem，要求 kind=="cli" 且 RootDir 存在。
2. 加载现有 manifest（拿 `Executable`）。
3. 构造一个 isolated env（复用 `cli.BuildEnv` shared 模式即可，因为只跑 --help），调 `cli.ProbeHelp(ctx, manifest.Executable, env)`。
4. 读 `package.json` 拿 `PackageMeta`。
5. 通过 provider 服务取当前**默认聊天 provider + 默认模型**（参考 `service/provider` 里 GetDefault 或类似方法；缺失则报 `ErrCliManifestGenerateFailed`）。
6. 调 `cli.Generate(ctx, GenerateParams{..., Caller: func(ctx, sys, usr) (string, error) { return agent.OneshotComplete(ctx, OneshotRequest{...}).Text, nil }})`。
7. 把返回的 manifest 写回 `manifest.json`（用 cli.SaveManifest）。
8. 同步 ExtensionItem.Tools（投影自 manifest.Tools，见 Task 5）。
9. 返回更新后的 ExtensionItem。

### 测试

- 用 `t.Setenv("LEMONTEA_DATA_DIR", ...)` + `withFakeCliManager`（复用 P3 的 fixture），再 monkey-patch 一个假的 `Generate` 入口（重构 Plugin 让生成器调用走一个可注入的字段）。
- 验证：调用成功后 manifest.json 内容更新、ExtensionItem.Tools 反映新 tools。
- 验证：provider 默认配置缺失时返回 `ErrCliManifestGenerateFailed`。

## Task 4: ToolSet adapter — cli.Manifest.Tools → agenttool.ToolSet

**Files:**
- Create: `backend/pkg/cli/tools_adapter.go`
- Create: `backend/pkg/cli/tools_adapter_test.go`

### 调研先行

阅读 `backend/pkg/mcp/manager.go` 的 `BuildToolSet` 实现，了解 `agenttool.ToolSet` 接口的形状（trpc-agent-go 的 `tool` 包）。CLI 的 ToolSet 不能像 MCP 那样依赖 stdio MCP server；我们要**自己实现** ToolSet 接口，返回一个动态构造的 tool 列表，每个 tool 的执行函数调 `cli.Manager.RunTool`。

参考 `function.FunctionTool[Input,Output]`（在 `tools/shell.go`）—— 它需要编译时已知的输入/输出 struct。CLI 的输入 schema 是 JSON Schema，运行时才知道；所以不能用 `FunctionTool`。需要直接实现 `agenttool.Tool` 接口（最低限度：`Name()`, `Description()`, `Schema()`, `Call(ctx, args json.RawMessage) (json.RawMessage, error)`）。具体接口签名以 trpc-agent-go 实际为准。

### 接口

```go
// BuildToolSet creates an agent ToolSet for one enabled CLI extension.
// All enabled tools in the manifest are surfaced; their Call() forwards to mgr.RunTool.
// enabledUserTools is the per-conversation enabled list; tools not in it are filtered out
// (consistent with how mcp.BuildToolSet does filtering).
func BuildToolSet(mgr *Manager, item data_models.ExtensionItem, manifest Manifest, enabledUserTools []string) (agenttool.ToolSet, error)
```

工具命名规范：`cli_{sanitized_extension_name}_{sanitized_tool_name}`（与 P2 spec 的 `cli_{name}_{tool_name}` 对齐；sanitize 复用 `mcpToolSetName` 同款逻辑或抽到 dir/util 公用 helper）。

### 测试

- 用一个假的 `agenttool.Tool` 通过反射或接口断言验证 `BuildToolSet` 返回的 ToolSet 内：tool 数量 = manifest.tools 中 enabled 数 ∩ enabledUserTools。
- Mock `cli.Manager.RunTool`（注入一个 `RunTool` 函数指针让测试替换），验证 ToolSet 的 Call → RunTool → 把 RunResult JSON 化作为 tool 输出。
- 验证 `requires_confirm: true` 的元数据被正确标记（具体如何传到 agent registry 取决于现有审批面板的接入点 —— 阅读 `tool_confirm.go` 与 builtin shell 工具的 `RequiresConfirm` 设置后再实现）。

### agent.Manager.buildCliToolSets

镜像现有 `buildMCPToolSets`：

```go
func (m *Manager) buildCliToolSets(enabledUserTools []string) []agenttool.ToolSet {
	config, err := loadAgentConfig()
	if err != nil || len(config.Extensions) == 0 {
		return nil
	}
	mgr, err := newCliManagerFromRuntime() // helper that mirrors plugin svc.resolveCliManager but is agent-side
	if err != nil {
		return nil
	}
	sets := make([]agenttool.ToolSet, 0)
	for _, item := range config.Extensions {
		if item.Kind != "cli" || !item.Enabled {
			continue
		}
		manifest, _ := pkgcli.LoadManifest(item.ConfigFilePath)
		ts, err := pkgcli.BuildToolSet(mgr, item, manifest, enabledUserTools)
		if err == nil && ts.Name() != "" {
			sets = append(sets, ts)
		}
	}
	return sets
}
```

在 `GetOrCreateRunner` 的 `llmagent.WithToolSets(...)` 调用处合并 mcp + cli：

```go
llmagent.WithToolSets(append(m.buildMCPToolSets(enabledUserTools), m.buildCliToolSets(enabledUserTools)...)),
```

### syncExtensionRuntime cli 投影

P3 设的 cli 分支当前是 `item.Tools = []`。改成读 manifest 把 `manifest.Tools` 投影成 `data_models.ExtensionTool`：

```go
if item.Kind == "cli" {
	manifest, err := pkgcli.LoadManifest(item.ConfigFilePath)
	if err != nil || manifest.Executable == "" {
		item.RuntimeStatus = "error"
		item.RuntimeMessage = "manifest missing; reinstall"
		item.Tools = []data_models.ExtensionTool{}
		return item
	}
	tools := make([]data_models.ExtensionTool, 0, len(manifest.Tools))
	for _, t := range manifest.Tools {
		tools = append(tools, data_models.ExtensionTool{
			ToolID:          "cli_" + sanitize(item.Name) + "_" + sanitize(t.Name),
			ServerID:        item.ID,
			Name:            t.Name,
			Description:     t.Description,
			Enabled:         t.Enabled,
			RequiresConfirm: t.RequiresConfirm,
		})
	}
	item.Tools = tools
	item.RuntimeStatus = "ready"
	item.RuntimeMessage = ""
	return item
}
```

## 难点与未决项

1. **trpc-agent-go ToolSet 接口形状**：Task 4 需先调研。MCP 的 BuildToolSet 是参考实现。如果接口要求编译时已知 input/output struct（像 `function.FunctionTool[I,O]`），需找到替代路径——可能用 `function.NewFunctionTool` 配 `interface{}` / `json.RawMessage` 输入。最差情况：把 cli 工具的 input/output 都强制走 `json.RawMessage`，类型由 schema 在 prompt 层告诉模型即可。
2. **agent → plugin/runtime 跨包依赖**：Task 5 step 0 提到把 `LoadPersistedState` 提到独立小包 `pkg/runtime_state/`。如果该重构面太大，临时方案是 agent 包通过 `pkg/dir` 直接读 state.json，但要复制 RuntimeState 结构体定义——优先重构，重复代码是技术债。
3. **manifest 生成质量**：当前 prompt 写得比较简单，遇到 `--help` 输出格式诡异（多级 ANSI 色码、子命令分层很深）的 CLI 可能 hallucinate。可在 Task 2 测试里加几个不同 CLI 的 --help 样本做端到端验证（pino-pretty、gh、aws cli 的截断版）。
4. **默认 provider/model 获取路径**：Task 3 需要从 provider 服务读「当前默认 chat provider + model」。若现有 API 不存在，可能要新增 `provider.GetDefaultChat()`；先调研后决定。
