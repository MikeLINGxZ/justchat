# Lemontea 基础设置设计文档

## 目标

实现一个独立的设置窗口，在点击侧边栏底部“设置”后打开，并完整覆盖以下范围：

- 通用设置：显示设置、语言与地区、文件
- 模型供应商设置：供应商列表、详情编辑、默认供应商、启用状态、删除
- 关于页面：应用信息展示

本次实现要求前后端打通，通用设置落盘到配置文件，供应商数据使用 gorm 存储，文件目录变更支持自动迁移数据并重启应用。

---

## 约束与前提

- 保持现有独立窗口方案，沿用 `Window.OpenSettings()` 打开 `window_settings` 窗口。
- 遵循现有仓库结构，不做无关重构。
- 所有界面必须支持浅色、深色主题。
- 所有设置文案必须支持简体中文、英文。
- 字体大小必须支持 5 档：极小、小、标准、大、超大。
- Go 中新增函数都要有注释，复杂逻辑和关键流程要在关键位置补充说明。
- 供应商新增入口仅允许选择内置 `ProviderType`：
  - `deepseek`
  - `aliyun`
  - `openrouter`
  - `ollama`
  - `openai_compatibility`

---

## 用户体验目标

设置窗口采用左右两栏基础布局：

- 左侧：一级设置菜单 `通用 / 模型供应商 / 关于`
- 右侧：根据当前一级菜单，显示“子菜单 + 详情区”或“直接内容区”

右侧支持两种内容模式：

1. `setting_content_child_menu + setting_content_child_content`
   - 用于 `通用` 和 `模型供应商`
   - 左边显示当前一级菜单的子菜单或供应商列表
   - 右边显示当前子项的详细内容

2. `setting_content_direct`
   - 用于 `关于`
   - 右侧整块直接展示内容

整体风格延续当前应用的现代化桌面界面风格，使用现有主题变量和字体大小机制，不单独引入新的视觉系统。

为了便于后续扩展更多设置项，本次设置界面前端按“通用设置框架 + 业务设置内容”的方式组织。通用布局、字段排版、操作栏和 dirty 状态能力抽成可复用组件，具体设置项只实现自己的字段和提交逻辑。

## 范围分解

### 通用

#### 显示设置

- 提供 5 档字体大小设置
- 支持在设置页内实时预览
- 提供“恢复默认”和“应用”按钮
- 仅当草稿值和当前已生效值不一致时，`应用` 按钮可点击
- 点击“应用”后：
  - 更新配置文件
  - 更新前端全局字体大小状态
  - 让当前设置窗口和主窗口立即生效

#### 语言与地区

- 提供“地区”和“语言”两个下拉框
- 提供“应用”按钮
- 仅当草稿值和当前已生效值不一致时，`应用` 按钮可点击
- 点击“应用”后：
  - 更新配置文件
  - 语言立即切换
  - 设置窗口与主窗口文案同步切换
- 本轮仅持久化“地区”设置，不附带更重的时区、格式化业务逻辑

#### 文件

- 显示当前数据目录
- 支持用户选择新的数据存储目录
- 提供“应用”按钮
- 点击“应用”后：
  - 弹出确认
  - 自动迁移当前配置和数据库到新目录
  - 切换应用数据目录
  - 重启应用
- 迁移失败时必须保留原目录并给出错误提示

### 模型供应商

- 使用子菜单布局
- 左侧子菜单区域显示供应商列表
- 左侧头部右侧提供新增按钮
- 新增时只能从内置供应商类型中选择
- 每个供应商列表项显示：
  - 图标
  - 名称
  - Base URL
  - 启用/禁用开关
  - 更多菜单：设为默认、删除
- 默认供应商项显示“默认”标签
- 右侧详情页显示：
  - 标题：`配置 {供应商名称}`
  - 提示文案：API 密钥仅本地加密保存
  - 启用状态开关
  - 表单字段：供应商名称、API 密钥、API 基础 URL
  - 模型列表标题区：模型数量、刷新按钮、添加模型按钮
  - 操作按钮：保存、测试连接、删除

### 关于

- 使用直接内容布局
- 顶部显示应用图标、名称 `Lemontea`、版本号 `v0.0.1-dev`
- 下方显示应用说明文字
- 保留“功能特性”“联系我们”的后续扩展空间

## 架构方案

本次采用“前端主导状态 + 后端提供持久化与系统能力”的方案。

### 前端职责

- 负责设置窗口布局、菜单切换和交互反馈
- 负责表单草稿态、dirty 状态、当前选中项
- 负责在“应用”或“保存”成功后同步全局运行态
- 负责本地预览和状态联动

### 后端职责

- 负责读写通用配置文件
- 负责供应商数据的增删改查
- 负责默认供应商切换
- 负责数据目录迁移
- 负责应用重启
- 负责向前端返回适合渲染的视图模型

### 数据分层原则

- `backend/models/data_models`
  - 只存真实落盘的数据结构
  - 包括配置文件结构、gorm 模型
- `backend/models/view_model`
  - 存前端需要的显示模型
  - 例如 `is_default`、`model_count`、`provider_icon`

## 前端设计

### 页面入口

保持单个前端构建产物，通过 URL 参数区分入口：

- 主窗口：`/`
- 设置窗口：`/?entry=settings&tab=about`

`App.tsx` 读取查询参数，渲染：

- `MainLayout`：主窗口
- `SettingsApp`：设置窗口

### 通用组件边界

本次设置页前端通用层按以下职责拆分：

- `SettingsShell`
  - 负责整个设置窗口的一级菜单布局、窗口级标题区、内容区切换
- `SettingsSectionLayout`
  - 负责右侧两种内容模式：
    - 子菜单 + 详情区
    - 直接内容区
- `SettingsSubmenuList`
  - 负责子菜单列表、供应商列表这类左侧选择区的统一样式与交互
- `SettingsPanelHeader`
  - 负责设置内容区的标题、描述、副操作区域
- `SettingsActionBar`
  - 负责“应用 / 保存 / 恢复默认 / 删除”等底部操作按钮区域
- `SettingsFieldRow`
  - 负责 label、说明文本、表单控件的统一排版
- `SettingsDirtyGuard`
  - 负责 dirty 状态、按钮启用态、离开提醒等通用行为

业务组件只负责自身字段和业务交互，例如：

- `DisplaySettingsView`
- `LocaleSettingsView`
- `FileSettingsView`
- `ProviderDetailView`
- `AboutSettingsView`

这样后续新增“账户”“快捷键”“实验功能”等设置项时，可以直接复用通用框架，而不需要重复实现整套布局和底部交互。

### 状态设计

前端状态分两层：

1. `appStore`
   - 存放已生效的全局运行态
   - 例如 `theme`、`fontSize`、`language`

2. `settingsStore`
   - 只服务设置窗口
   - 存放：
     - 一级菜单
     - 当前子菜单
     - 当前选中供应商
     - 各设置项草稿值
     - 后端加载结果
     - 请求中状态
     - 错误提示状态

核心原则：

- “草稿值”与“已生效值”必须分离
- “列表展示值”与“详情页编辑草稿”必须可同步但不互相覆盖错误

### 交互规则

#### 显示设置

- 改变字体大小时，预览区域立即响应
- 主应用字体不立即变化
- 点击“应用”后，更新配置并同步主应用运行态

#### 语言与地区

- 下拉变更后仅更新草稿
- 点击“应用”后统一提交
- 成功后立即调用 i18n 切换语言

#### 文件

- 路径变更后按钮可点击
- 点击“应用”先弹确认框
- 成功后由后端发起重启，不要求前端继续做复杂恢复

#### 供应商

- 列表开关与详情开关双向同步
- 设为默认后，列表和详情同时刷新
- 删除默认供应商时阻止直接删除，并给出提示
- 新增供应商后自动切换到该供应商详情页

## 后端设计

### 新增服务

本次将后端服务拆分为两个独立服务文件：

```text
backend/service/settings.go
backend/service/provider.go
```

`settings.go` 对前端暴露以下能力：

- 获取设置窗口初始化数据
- 保存显示设置
- 保存语言与地区设置
- 迁移数据目录
- 读取应用版本信息
- 重启应用

`provider.go` 对前端暴露以下能力：

- 获取供应商列表
- 获取单个供应商详情
- 创建供应商
- 更新供应商
- 删除供应商
- 设置默认供应商
- 测试供应商连接
- 刷新供应商模型列表
- 新增自定义模型

### 配置文件模型

现有 `backend/models/data_models/config.go` 本次扩展为至少覆盖：

- `Locale`
- `Language`
- `FontSize`
- `DataDir`
- `LogLevel`
- `DefaultProviderID`

本次不把“主题”纳入设置窗口的通用设置，也不把它并入后端配置文件范围。主题继续沿用现有主窗口快捷菜单和前端运行态机制；设置窗口只需要正确响应当前主题即可。

### 自定义数据目录定位信息

当用户把数据目录切换到自定义路径后，应用仍然需要在用户 Home 目录下的默认 `.lemontea` 目录中保留一份“目录定位信息”，用于下次启动时找到真正的数据目录。

本次增加一个固定定位文件，例如：

```text
~/.lemontea/data_dir.json
```

该文件只负责记录当前生效的数据目录路径，不承载完整业务配置。启动时 `backend/pkg/dir/dir.go` 按以下优先级解析数据目录：

1. 环境变量 `LEMONTEA_DATA_DIR`
2. `~/.lemontea/data_dir.json` 中记录的自定义目录
3. 默认 `~/.lemontea`

这样可以保证：

- 用户迁移到自定义目录后，应用重启仍能定位到新的配置文件和数据库
- 即使真实数据目录被切换，Home 下仍保留一个稳定入口
- 数据目录迁移逻辑不依赖硬编码或一次性内存状态

### 日志能力

本次新增日志包：

```text
backend/pkg/logger
```

日志能力要求如下：

- 配置文件 `config.json` 新增 `log_level` 字段
- 运行时只写入“级别大于等于配置级别”的日志
- 日志目录固定在当前生效数据目录下的 `logs/`
- 日志文件按日期拆分，例如：

```text
{data_dir}/logs/2026-05-10.log
```

- 应用启动时初始化日志组件
- 当数据目录切换后，重启后的新进程必须自动把日志写到新目录下
- 日志库应提供统一入口，供设置服务、供应商服务、目录迁移逻辑等后端模块复用

日志级别本轮至少支持：

- `debug`
- `info`
- `warn`
- `error`

其中：

- 当前日志级别为 `info` 时，写入 `info`、`warn`、`error`
- 当前日志级别为 `warn` 时，写入 `warn`、`error`
- 当前日志级别为 `error` 时，只写入 `error`

### 供应商模型

沿用 `backend/models/data_models/provider.go`，但要补齐视图层输出能力。

前端不直接依赖数据库模型，由后端返回类似结构：

```go
type ProviderItem struct {
    ID          uint   `json:"id"`
    ProviderName string `json:"provider_name"`
    ProviderType string `json:"provider_type"`
    BaseURL     string `json:"base_url"`
    Enabled     bool   `json:"enabled"`
    IsDefault   bool   `json:"is_default"`
    ModelCount  int    `json:"model_count"`
    Icon        string `json:"icon"`
}
```

### 数据目录迁移

迁移逻辑需要独立封装，流程如下：

1. 校验目标目录可写
2. 创建目标目录
3. 复制当前 `config.json`
4. 复制当前数据库文件
5. 确保目标目录下 `logs/` 目录可创建
6. 更新目标目录中的配置文件 `DataDir`
7. 在 `~/.lemontea/data_dir.json` 写入新的目录定位信息
8. 校验定位文件和目标目录内容都已落盘
9. 触发应用重启

失败处理要求：

- 任一中间步骤失败都返回错误
- 不删除旧目录数据
- 不覆盖原有可用的目录定位信息
- 不把运行中的数据目录切换到半完成状态

## 数据流设计

### 设置窗口初始化

1. 打开设置窗口
2. 前端读取 URL 参数确定默认一级菜单
3. 调用后端初始化接口
4. 返回：
   - 配置数据
   - 供应商列表
   - 当前默认供应商
   - 应用版本信息
   - 当前日志级别
5. 前端填充 `settingsStore`

### 应用显示设置

1. 用户修改草稿字体大小
2. 预览区域更新
3. 点击“应用”
4. 后端写入配置文件
5. 前端更新 `appStore.fontSize`
6. 设置窗口和主窗口立即生效

### 应用语言与地区

1. 用户修改语言/地区草稿
2. 点击“应用”
3. 后端写入配置文件
4. 前端切换 i18n 语言
5. 全局文案刷新

### 迁移数据目录

1. 用户选择新目录
2. 点击“应用”
3. 前端弹确认
4. 后端执行迁移
5. 成功后重启应用
6. 失败则提示错误并保持当前状态

### 供应商详情编辑

1. 用户在左侧选择供应商
2. 右侧详情显示当前配置
3. 用户编辑草稿
4. 点击“保存”
5. 后端更新 gorm 数据
6. 前端刷新列表和详情

## 错误处理

### 前端

- 所有保存、迁移、删除、测试连接操作都要有 loading 态
- 错误反馈统一使用当前项目已有提示方式；若仓库里还没有统一消息组件，则可先在设置窗口内使用轻量级提示条

### 后端

- 所有对外暴露方法都返回明确错误
- 数据迁移、默认供应商切换、删除供应商等关键路径应做保护性校验
- API Key 不在日志中打印明文

## 实施顺序

本次按以下顺序推进：

1. 扩展配置模型与后端设置服务骨架
2. 接入设置窗口前端入口与基础布局
3. 完成通用设置前后端打通
