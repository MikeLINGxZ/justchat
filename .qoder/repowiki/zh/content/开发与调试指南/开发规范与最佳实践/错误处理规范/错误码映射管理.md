## 引言
在前后端分离的架构中，错误信息的统一管理是提升用户体验和开发效率的关键环节。本规范旨在说明如何维护后端 `backend/utils/ierror/code.go` 中定义的 `ErrorCode` 枚举值与前端 `frontend/src/utils/errorHandler.ts` 中 `ERROR_MESSAGE_MAP` 的语义一致性，确保用户在前端看到的错误提示准确、友好且与后端逻辑一致。

## 前后端错误码结构分析

### 后端错误码定义
后端通过 `ierror` 包统一管理错误码，所有错误码以常量形式定义在 `backend/utils/ierror/code.go` 文件中，类型为 `ErrorCode`（字符串别名）。每个错误码均配有中文注释说明其语义。

```go
type ErrorCode string

const (
	// ErrCodeInternalError 系统内部错误
	ErrCodeInternalError ErrorCode = "ErrCodeInternalError"
	// ErrCodeUsernameExists 用户名已被使用
	ErrCodeUsernameExists = "ErrCodeUsernameExists"
	// ...
)
```

这些错误码通过 `ierror.New()` 函数创建错误实例，并在服务层返回给调用方。

### 前端错误消息映射
前端在 `frontend/src/utils/errorHandler.ts` 中维护一个静态映射表 `ERROR_MESSAGE_MAP`，将后端返回的错误码字符串映射为用户可读的中文提示。

```typescript
const ERROR_MESSAGE_MAP: Record<string, string> = {
  'ErrCodeInvalidAccountPassword': '用户名或密码错误',
  'ErrCodeAccountNotFound': '账户不存在',
  'ErrCodeUsernameExists': '用户名已存在',
  // ...
};
```

当接收到后端错误响应时，`extractErrorMessage` 函数会查找此映射表，返回对应的友好消息。

## 错误码语义一致性维护机制
为确保前后端错误语义一致，必须建立严格的协同流程。当前系统存在部分不一致现象，例如：
- 后端定义了 `ErrCodeModelNotFound`（模型不存在），但前端 `ERROR_MESSAGE_MAP` 中无对应条目。
- 后端定义了 `ErrCodeChatNotFound`（对话不存在），前端也未注册。
- 前端定义了 `ErrCodeAccountDisabled`、`ErrCodePasswordExpired` 等错误码，但后端未找到对应定义。

这种不一致可能导致用户看到默认错误消息而非精准提示，影响体验。

## 动态扩展机制：addErrorMapping
为应对插件化或动态配置场景，前端提供了 `addErrorMapping` 接口，允许在运行时动态注册新的错误码映射：

```typescript
export const addErrorMapping = (errorCode: string, message: string): void => {
  ERROR_MESSAGE_MAP[errorCode] = message;
};
```

### 应用价值
1. **插件系统支持**：第三方插件可在加载时调用 `addErrorMapping` 注册自身特有的错误码，无需修改核心代码。
2. **动态配置**：通过配置中心下发错误码映射，前端可动态更新提示内容，实现多语言或运营文案调整。
3. **灰度发布**：新功能的错误码可在灰度阶段通过动态方式注入，避免全量发布风险。
4. **热修复**：发现错误提示不准确时，可通过脚本动态修正，无需重新构建部署。

该机制增强了系统的灵活性和可扩展性。

## 新增错误码标准流程
为确保错误码双端语义一致，应遵循以下标准化流程：

1. **后端定义**：在 `backend/utils/ierror/code.go` 中添加新的 `ErrorCode` 常量，并附上中文注释。
   ```go
   // ErrCodeNewFeatureError 新功能特有错误
   ErrCodeNewFeatureError = "ErrCodeNewFeatureError"
   ```

2. **后端使用**：在服务逻辑中通过 `ierror.New(ierror.ErrCodeNewFeatureError)` 抛出该错误。

3. **前端注册**：在前端适当位置（如模块初始化时）调用 `addErrorMapping` 注册中文提示：
   ```typescript
   addErrorMapping('ErrCodeNewFeatureError', '新功能出现异常，请稍后重试');
   ```

4. **验证映射**：通过 `getErrorMappings()` 接口或调试工具确认映射已生效。

此流程确保了错误码从定义到展示的全链路一致性。

## 调试接口：getErrorMappings
前端提供了 `getErrorMappings()` 调试接口，用于获取当前完整的错误映射表：

```typescript
export const getErrorMappings = (): Record<string, string> => {
  return { ...ERROR_MESSAGE_MAP };
};
```

### 开发期排查作用
- **检查遗漏**：开发人员可在浏览器控制台调用 `getErrorMappings()`，检查新添加的错误码是否已注册。
- **验证动态注册**：确认插件或动态脚本是否成功注入了新的错误映射。
- **对比后端**：将返回结果与后端 `code.go` 中的定义进行对比，快速发现不一致项。
- **调试工具集成**：可集成到开发者面板中，作为诊断工具的一部分。

该接口是保障映射完整性的关键调试手段。

## 自动化检查建议
为防止人为疏忽导致映射缺失，建议建立自动化检查机制：

1. **CI/CD 钩子**：在代码提交或合并请求时，运行脚本自动扫描 `backend/utils/ierror/code.go` 中的所有 `ErrorCode` 常量，并验证其是否在 `frontend/src/utils/errorHandler.ts` 的 `ERROR_MESSAGE_MAP` 或通过 `addErrorMapping` 被引用。

2. **检查脚本示例逻辑**：
   - 使用正则提取 `code.go` 中所有 `ErrCodeXXX` 常量名。
   - 搜索 `errorHandler.ts` 中是否包含这些常量名的字符串匹配（静态映射）或 `addErrorMapping` 调用（动态注册）。
   - 若发现未映射的错误码，中断构建并提示警告。

3. **定期审计**：在每日构建中运行完整性检查，生成映射覆盖率报告。

4. **IDE 插件**：开发编辑器插件，在定义新错误码时提示开发者前往前端注册。

自动化检查能从根本上杜绝映射遗漏问题。

## 总结
维护前后端错误码语义一致性是保障用户体验的重要基础。通过遵循“后端定义 → 前端注册”的标准流程，利用 `addErrorMapping` 实现动态扩展，并借助 `getErrorMappings` 进行调试验证，可有效管理错误信息。建议引入自动化检查脚本，将一致性保障纳入研发流程，避免人为遗漏，提升系统健壮性与可维护性。
