## 简介
本项目采用清晰、一致的命名约定，以提升代码可读性和可维护性。Go后端使用驼峰式命名法，接口方法采用动词+名词结构；TypeScript前端遵循React社区规范，组件名使用大驼峰，Hook函数以`use`开头；Wails RPC服务注册名称与Go方法名保持一致，确保跨语言调用的可读性。通过具体代码示例对比展示符合规范与违反规范的命名方式，强调命名一致性在团队协作和长期维护中的重要性。

## Go后端命名规范

Go后端采用驼峰式命名法（CamelCase），接口方法使用动词+名词结构，变量名简洁达意。这种命名方式提高了代码的自解释能力，使开发者能够快速理解函数用途。

### 接口方法命名
接口方法采用动词+名词结构，清晰表达操作意图。例如：

- `ChatList`：获取聊天列表
- `ChatMessages`：获取聊天消息
- `Completions`：执行补全操作
- `DeleteChat`：删除聊天记录
- `RenameChat`：重命名聊天
- `CollectionChat`：收藏/取消收藏对话
- `GetModels`：获取模型列表
- `OpenSettingsWindow`：打开设置窗口

这些方法名直接反映了其功能，无需额外注释即可理解。

### 变量与结构体命名
变量名简洁达意，结构体名称使用大驼峰，字段名称使用小驼峰：

- `chatUuid`：聊天唯一标识
- `message`：消息对象
- `providerModel`：提供方模型
- `historyMessages`：历史消息列表
- `msgChan`：消息通道
- `errChan`：错误通道
- `doneChan`：完成信号通道

结构体如`Service`、`Storage`等均使用大驼峰命名，符合Go语言惯例。

## TypeScript前端命名规范

TypeScript前端遵循React社区规范，组件名使用大驼峰（PascalCase），Hook函数以`use`开头，确保代码风格统一且易于识别。

### 组件命名
React组件名称使用大驼峰命名法，明确标识其为UI组件：

- `MessageAction`：消息操作组件
- `ChatInput`：聊天输入组件（根据项目结构推断）
- `MessageList`：消息列表组件（根据项目结构推断）
- `ReasoningContent`：推理内容组件（根据项目结构推断）
- `AnimatedAlert`：动画提醒组件（根据项目结构推断）

组件文件路径与名称一致，如`MessageAction/index.tsx`，便于模块化组织。

### Hook函数命名
自定义Hook函数以`use`开头，符合React Hook命名约定：

- `useModels`：用于获取模型列表的Hook
- `useViewportHeight`：用于获取视口高度的Hook

```typescript
export const useModels = (params: UseModelsParams = {}): UseModelsReturn => {
  // Hook实现
};
```

该Hook返回模型列表、加载状态、错误信息及刷新函数，封装了与后端服务的交互逻辑。

## Wails RPC服务命名规范

Wails RPC服务注册名称与Go方法名保持一致，确保跨语言调用的可读性和一致性。前端通过自动生成的绑定文件调用后端方法，方法名完全对应。

### RPC方法映射
Wails自动生成前端绑定代码，Go方法名直接映射为TypeScript函数名：

```typescript
/**
 * AddProvider 添加供应商
 */
export function AddProvider(provider: view_models$0.Provider): $CancellablePromise<void> {
    return $Call.ByID(2363217024, provider);
}

/**
 * ChatList 聊天列表
 */
export function ChatList(offset: number, limit: number, keyword: string | null, isCollection: boolean): $CancellablePromise<view_models$0.ChatList | null> {
    return $Call.ByID(1229500932, offset, limit, keyword, isCollection).then(($result: any) => {
        return $$createType1($result);
    });
}

/**
 * GetModels 获取所有模型
 */
export function GetModels(): $CancellablePromise<view_models$0.Model[]> {
    return $Call.ByID(2435611672).then(($result: any) => {
        return $$createType7($result);
    });
}
```

### 命名一致性优势
- **可读性强**：前后端方法名一致，降低理解成本
- **维护性高**：修改后端方法时，前端调用点一目了然
- **减少错误**：避免因命名不一致导致的调用错误
- **自动化支持**：Wails自动生成绑定代码，确保命名同步

## 命名一致性对维护性的影响

命名一致性是代码可维护性的关键因素。本项目通过严格的命名规范，提升了代码的可读性、可搜索性和可维护性。

### 符合规范的命名示例
```go
func (s *Service) ChatList(offset, limit int, keyword *string, isCollection bool) (*view_models.ChatList, error)
func (s *Service) GetModels() ([]view_models.Model, error)
```

```typescript
export const useModels = (params: UseModelsParams = {}): UseModelsReturn
const MessageAction: React.FC<MessageActionProps>
```

### 违反规范的命名示例（反面教材）
```go
// ❌ 错误：动词位置不当，不符合习惯
func (s *Service) ListChat(...) 

// ❌ 错误：使用下划线命名法，不符合Go惯例
func (s *Service) get_models()

// ❌ 错误：变量名不达意
func (s *Service) Completions(chatId string, mdl string, msg schema.Message)
```

```typescript
// ❌ 错误：组件名未使用大驼峰
const messageAction = () => {}

// ❌ 错误：Hook未以use开头
const models = () => {}
```

### 维护性优势
1. **降低认知负荷**：开发者无需记忆不同命名规则
2. **提高搜索效率**：方法名具有可预测性，便于全局搜索
3. **减少bug**：命名一致减少调用错误
4. **便于重构**：统一命名模式支持批量重命名
5. **新人友好**：新成员能快速适应代码风格

## 结论
本项目建立了完整的命名规范体系：Go后端采用驼峰式命名法，接口方法使用动词+名词结构；TypeScript前端遵循React社区规范，组件名大驼峰，Hook函数以`use`开头；Wails RPC服务注册名称与Go方法名保持一致。这种跨语言的一致性命名策略显著提升了代码的可读性、可维护性和团队协作效率。建议在项目开发中严格执行此规范，并通过代码审查确保一致性。
