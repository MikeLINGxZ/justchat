## 简介
本项目旨在统一前后端代码风格，提升团队协作效率与代码可维护性。前端使用TypeScript，后端使用Go语言。通过自动化工具链确保所有提交代码均符合既定格式标准。本文档详细说明格式化规则、工具集成方式及开发环境配置建议。

## Go代码格式化规范
Go语言代码必须使用官方工具`gofmt`进行格式化。禁止手动调整缩进、空行或括号位置等格式细节。所有`.go`文件在提交前应自动通过`gofmt -w`格式化。

该规则适用于以下目录中的所有Go文件：
- `backend/models/`
- `backend/pkg/logger/`
- `backend/service/`
- `backend/storage/`
- `backend/utils/`
- `main.go`

## TypeScript代码格式化规范
TypeScript代码采用Prettier进行统一格式化，具体配置如下：

- 引号风格：单引号 `'`
- 缩进：2个空格
- 行尾逗号：启用（`"es5"`风格）
- 换行符：LF
- 文件末尾换行：启用
- 最大行宽：80字符

Prettier配置应通过项目根目录下的`.prettierrc`文件定义（若存在），并与ESLint规则兼容。当前项目已安装`prettier`及相关ESLint插件，确保格式化与 lint 检查协同工作。

## 格式化工具集成方案
项目已在`package.json`中定义标准化脚本命令，便于开发者执行格式化与检查操作。

### 支持的脚本命令
```json
"scripts": {
  "dev": "vite",
  "build:dev": "tsc && vite build --minify false --mode development",
  "build": "tsc && vite build --mode production",
  "lint": "eslint .",
  "preview": "vite preview"
}
```

建议扩展以下脚本以支持格式化：
```json
"scripts": {
  "format": "prettier --write \"src/**/*.{ts,tsx}\" \"bindings/**/*.{ts,tsx}\"",
  "format:check": "prettier --check \"src/**/*.{ts,tsx}\" \"bindings/**/*.{ts,tsx}\""
}
```

上述命令将对`src`和`bindings`目录下的所有TypeScript文件执行格式化。

## 编辑器自动格式化配置（VS Code）
建议开发者在VS Code中配置保存时自动格式化，避免手动执行格式化命令。

### 推荐的`.vscode/settings.json`配置
```json
{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.tabSize": 2,
  "prettier.singleQuote": true,
  "prettier.trailingComma": "es5",
  "prettier.printWidth": 80,
  "files.autoSave": "onFocusChange"
}
```

此配置确保：
- 保存文件时自动应用Prettier格式化
- 使用单引号而非双引号
- 启用末尾逗号
- 缩进为2个空格
- 文件在失去焦点时自动保存（可选）

## 预提交钩子（pre-commit）集成
为防止未格式化的代码被提交，建议集成`pre-commit`钩子，在每次提交前自动运行格式化检查。

### 使用Husky + lint-staged配置示例
1. 安装依赖：
```bash
npm install --save-dev husky lint-staged
```

2. 初始化Husky：
```bash
npx husky init
```

3. 添加`lint-staged`配置至`package.json`：
```json
"lint-staged": {
  "src/**/*.{ts,tsx}": [
    "prettier --write",
    "eslint --fix"
  ],
  "bindings/**/*.{ts,tsx}": [
    "prettier --write"
  ]
}
```

4. 创建`.husky/pre-commit`钩子：
```bash
npx lint-staged
```

此配置将在每次`git commit`时自动格式化变更的TypeScript文件，并修复可自动处理的ESLint问题。

## 最佳实践建议
- 所有开发者应在本地启用编辑器自动格式化功能
- 提交代码前务必运行`npm run format`确保一致性
- 若团队成员使用不同编辑器，应共享`.editorconfig`文件以统一基础格式
- CI/CD流水线中应包含`npm run format:check`步骤，防止不符合格式的代码合入主干

通过以上规范与工具链集成，可有效保障项目代码风格统一，减少代码审查中的格式争议，提升整体开发效率。
