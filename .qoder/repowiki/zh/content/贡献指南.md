## 分支管理策略

本项目采用双分支开发模型，确保代码的稳定性与开发的灵活性：

- `main` 分支：作为主干分支，始终保持稳定状态，仅用于发布版本。所有生产环境的构建均基于此分支。
- `dev` 分支：作为主要开发分支，所有新功能和修复均在此分支上进行集成。发布前会将 `dev` 分支合并至 `main`。

开发者不应直接向 `main` 或 `dev` 分支推送代码，而应通过特性分支（feature branch）发起 Pull Request 进行代码合并。

## 提交信息格式规范

所有提交信息必须遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范，以保证提交历史的可读性和自动化版本管理的准确性。

提交信息格式如下：
```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

常用类型包括：
- `feat`：新增功能
- `fix`：修复缺陷
- `docs`：文档更新
- `style`：代码格式调整（不影响逻辑）
- `refactor`：重构（既不新增功能也不修复缺陷）
- `test`：测试相关改动
- `chore`：构建过程或辅助工具变动

示例：
```
feat(chat): 添加消息流式渲染功能
fix(storage): 修复消息增量更新时内容拼接错误
docs(readme): 更新开发环境搭建说明
```

## 代码风格与格式化

### Go 代码
- 使用 `gofmt` 进行代码格式化
- 所有 Go 源文件应位于 `backend/` 目录下
- 日志输出应通过 `backend/pkg/logger` 包统一处理，确保日志格式一致性

### TypeScript 代码
- 使用 **ESLint + Prettier** 统一代码风格
- 前端代码位于 `frontend/src/` 目录下
- 所有组件和工具函数应遵循 React 函数式组件规范
- 类型定义应集中管理于 `frontend/types/index.ts` 或对应模块的类型文件中

## Pull Request 流程

标准的 Pull Request 流程如下：

1. **Fork 仓库**：点击 GitHub 仓库页面的 "Fork" 按钮创建个人副本
2. **克隆到本地**：
   ```bash
   git clone https://github.com/your-username/lemon_tea_desktop.git
   ```
3. **创建特性分支**：
   ```bash
   git checkout -b feature/your-feature-name dev
   ```
4. **进行修改并提交**：
   ```bash
   git add .
   git commit -m "feat: add your feature"
   ```
5. **推送分支**：
   ```bash
   git push origin feature/your-feature-name
   ```
6. **发起 Pull Request**：
   - 访问 GitHub 仓库页面
   - 点击 "Compare & pull request"
   - 基于 `dev` 分支创建 PR
7. **等待 CI 通过与代码审查**：
   - 确保所有 CI 检查通过（由 `wails3 dev` 和构建脚本保障）
   - 根据审查意见修改代码
   - 合并后由维护者集成至 `dev` 分支

## 测试与文档要求

鼓励贡献者在提交功能或修复时：

- **补充测试用例**：对于核心逻辑（如消息处理、存储操作），应提供单元测试或集成测试
- **更新相关文档**：包括但不限于：
  - `README.md`：项目说明与使用指南
  - `frontend/doc/README.md`：前端实现细节
  - API 接口文档（如 `service.ts` 中的方法说明）

特别是涉及用户交互的功能（如聊天消息流式渲染、模型选择等），应在文档中清晰说明其行为逻辑。
