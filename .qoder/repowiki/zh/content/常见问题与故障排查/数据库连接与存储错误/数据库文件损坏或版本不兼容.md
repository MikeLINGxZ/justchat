## 引言
本文件旨在说明在使用 Wails 桌面应用过程中，SQLite 数据库可能因意外断电、强制关闭或跨版本升级而导致数据库文件损坏或格式不兼容的问题。文档将介绍如何通过命令行工具检测数据库完整性，分析数据模型变更对旧数据的影响，并提供多种数据恢复策略。

## 数据库损坏的常见原因

SQLite 数据库文件损坏通常由以下几种情况引起：

- **意外断电或程序崩溃**：当应用程序正在写入数据库时发生断电或崩溃，可能导致事务未完成，进而破坏数据库结构。
- **强制关闭应用**：用户强制终止进程可能导致写操作中断，造成数据库页损坏。
- **跨版本升级导致的结构不兼容**：新版本应用中 `models.go` 定义的数据结构发生变化（如新增非空字段），而旧数据库中无对应字段，引发读取失败。

此类问题可能导致应用启动失败或数据无法加载，典型错误信息包括 `"database disk image is malformed"`。

## 验证数据库完整性的方法

可通过 `sqlite3` 命令行工具验证数据库文件的完整性。具体步骤如下：

1. 打开终端并进入数据库所在目录（默认路径为 `~/lemontea/data.db`）。
2. 执行以下命令打开数据库：
   ```bash
   sqlite3 data.db
   ```
3. 在 SQLite 提示符下运行完整性检查命令：
   ```sql
   PRAGMA integrity_check;
   ```
4. 若返回结果为 `ok`，表示数据库结构完整；若返回错误信息如 `database disk image is malformed` 或 `rowid not unique`，则表明数据库已损坏。

此外，还可使用 `PRAGMA quick_check;` 进行快速检查，适用于大型数据库。

## 实体变更导致的数据兼容性问题

当新版本应用更新了数据模型（如 `backend/models/data_models/models.go` 中的结构体定义），可能会破坏与旧数据库的兼容性。常见场景包括：

- **新增非空字段但无默认值**：若新版本 `Message` 结构体新增一个非空字符串字段（如 `source string`），而旧数据库中该字段缺失，GORM 在映射时会报错。
- **字段类型变更**：将整型字段改为字符串类型，可能导致类型转换失败。
- **表结构调整**：删除或重命名字段后未提供迁移逻辑，导致查询失败。

例如，在 `chat_message.go` 中，`SaveOrUpdateDeltaMessage` 方法依赖 `Message` 字段的嵌套结构进行增量更新。若该结构在数据库中不一致，会导致更新失败并抛出错误。

此类问题通常表现为应用无法读取历史聊天记录或保存新消息失败。

## 恢复方案与操作流程

针对数据库损坏或不兼容问题，可采取以下恢复措施：

### 方案一：从备份恢复数据库
若存在定期备份（如 `data.db.bak`），可直接替换损坏的数据库文件：
```bash
cp ~/lemontea/data.db.bak ~/lemontea/data.db
```

### 方案二：导出 JSON 聊天记录
若数据库仍可部分读取，可通过服务层导出有效数据。`chat.go` 中的 `ChatMessages` 方法可获取指定会话的消息列表，前端可调用此接口将数据导出为 JSON 文件，避免数据丢失。

### 方案三：重建数据库结构并迁移数据
1. **停止应用**，防止写入冲突。
2. **重命名旧数据库**：
   ```bash
   mv ~/lemontea/data.db ~/lemontea/data.db.corrupted
   ```
3. **启动应用**，触发 `AutoMigrate` 重建最新结构的数据库。
4. **编写迁移脚本**：从旧数据库中提取可用数据（如聊天标题、消息内容），插入新数据库对应表中。
5. **验证数据完整性**：使用 `PRAGMA integrity_check;` 确认新数据库无错误。

该流程需谨慎操作，建议在测试环境中先行验证。

## 结论

SQLite 数据库虽轻量可靠，但在异常情况下仍可能损坏。开发者应重视数据模型变更带来的兼容性风险，建议在版本升级时提供数据库迁移脚本。用户应定期备份数据库文件，以应对突发故障。通过 `PRAGMA integrity_check` 可快速诊断问题，结合数据导出与重建策略，可有效恢复关键信息。
