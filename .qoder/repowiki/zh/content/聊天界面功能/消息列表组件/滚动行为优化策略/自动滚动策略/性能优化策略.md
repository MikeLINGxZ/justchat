## 自动滚动性能优化

在聊天界面中，自动滚动功能需要在保证实时性的同时避免频繁的重排和重绘操作。系统通过精细化的滚动控制策略，在流式消息生成和普通消息更新场景下采用不同的滚动行为，确保性能和用户体验的平衡。

## requestAnimationFrame的应用

`requestAnimationFrame`被用于将滚动操作推迟到下一动画帧执行，有效减少浏览器的重排和重绘开销。当检测到新消息或流式更新时，系统不会立即执行滚动，而是将其安排在下一个合适的渲染时机。

```mermaid
flowchart TD
A["检测到新消息或流式更新"] --> B{"是否启用自动滚动?"}
B --> |是| C["使用requestAnimationFrame调度滚动"]
B --> |否| D["跳过滚动操作"]
C --> E["在下一帧执行滚动"]
E --> F["避免当前帧的重排重绘"]
F --> G["提升整体渲染性能"]
```

## 被动事件监听器的作用

被动事件监听器（passive event listeners）被应用于滚动相关的事件监听，显著提升了滚动事件的响应速度。通过设置`{ passive: true }`选项，系统告知浏览器该事件监听器不会调用`preventDefault()`，从而允许浏览器提前处理滚动操作。

```mermaid
sequenceDiagram
participant 用户
participant 浏览器
participant 事件监听器
用户->>浏览器 : 开始滚动操作
浏览器->>事件监听器 : 触发wheel/touch事件
事件监听器-->>浏览器 : 立即响应无需等待
浏览器->>用户 : 平滑滚动无延迟
事件监听器->>浏览器 : 异步处理业务逻辑
```

## 动态滚动模式选择

系统基于`hasStreamingMessage`状态动态选择滚动模式，实现流式生成和普通更新场景的差异化处理。

```mermaid
flowchart LR
A["消息更新事件"] --> B{"是否存在流式消息?"}
B --> |是| C["使用即时滚动<br>scrollToBottomInstant"]
B --> |否| D["使用平滑滚动<br>scrollToBottomSmooth"]
C --> E["保证实时性<br>无动画效果"]
D --> F["提升用户体验<br>平滑动画效果"]
```

## 用户体验优化

通过智能的滚动策略和状态管理，系统在各种使用场景下都能提供流畅的用户体验。首次进入、消息发送、AI生成等场景都经过专门优化。

